## 단계 1 페어 프로그래밍 회고

### 페어와 합의한 부분

#### 1. 계층별 메서드 네이밍 컨벤션

각 계층에서 호출하는 메서드의 이름을 어떻게 짓는 게 적절한지 헷갈렸다.

- Controller: `addTheme`
- Service: `addTheme`
- Repository: `insert` / `create` / `save` 중 무엇?

계층이 아래로 내려가면서 역할이 달라지는데(Repository는 DB, Service는 비즈니스/트랜잭션) 그 역할에 맞는 네이밍을 찾는 일이 쉽지 않았다.

**합의**: Controller와 Service는 동일한 메서드명을 사용하고, Repository는 CRUD라는 표준 개념이 있으니 그 안에서 가져다 쓰기로 했다.

------

#### 2. 검증의 위치 — DTO에서 할 것인가, Model에서 할 것인가

원래는 Theme Model에서 이름 길이를 검증하고 있었다. 그런데 흐름을 따라가 보니, 클라이언트의 입력이 RequestBody로 들어와 `ThemeRequest`(DTO)로 만들어지는 시점에는 검증이 일어나지 않고, Service 레이어에서 DTO가 Theme으로 변환될 때 비로소 `IllegalArgumentException`이 터지는 구조였다.

이 흐름이 맞는지 직접 확인해 보기로 했다. DTO에서 검증한 경우와 하지 않은 경우의 차이를 비교했다.

| 검증 위치  | 결과                                                         |
| ---------- | ------------------------------------------------------------ |
| DTO 검증 X | DB에 저장은 되어버린다. 이후 Entity 변환에서 실패하면서 클라이언트는 `500 Server Error`를 받는다. |
| DTO 검증 O | DB에 저장되지 않고, 클라이언트는 `400 Client Error`를 받는다. 우리가 원했던 흐름이다. |

**합의**: DTO에서 검증을 추가하기로 했다.

------

#### 3. DTO를 모든 레이어에서 사용하는 것

DTO가 Controller -> Service -> Repository에 걸쳐 모두 사용되고 있었다. 모든 레이어가 DTO를 의존하는 구조가 처음에는 어색하게 느껴졌지만, 다시 생각해 보면 "계층 간 데이터 전달"이라는 DTO 본연의 책임을 적절히 활용하는 것 아닌가 싶었다.

**합의**: 문제가 없어 보이므로 일단 이 구조로 구현을 진행한다.

------

#### 4. Theme과 ThemeResponse의 getter가 많은 것

Theme과 ThemeResponse가 사실상 동일한 구조라 좀 어색했고, getter가 많다는 점도 마음에 걸렸다. 다만 이 getter들은 상태 변경에 쓰이는 것이 아니라 단순히 값을 꺼내는 용도이므로 TDA(Tell, Don't Ask) 위반은 아니라고 정리했다.



#### 5. 존재하지 않는 ID 삭제 시 응답 — 400인가 204인가

원래 토리의 코드에서는 존재하지 않는 ID에 대한 DELETE 요청을 서버에서 예외로 처리해 클라이언트에게 `500 Server Error`를 내려주고 있었다.

여기서 "500이 맞나?"라는 의문이 들었다. 존재하지 않는 리소스를 지워달라는 건 클라이언트 측의 잘못된 요청에 가깝다고 판단했고, 그래서 `400 Error`로 수정한 뒤 오류 메시지를 Response Body에 담아 클라이언트에 전달하는 방향으로 가려고 했다. 이 과정에서 전역 에러 핸들러인 `@RestControllerAdvice`를 활용해, 예외가 발생하면 정해진 `ErrorResponse` 형식에 예외 메시지를 담아 클라이언트에 전송하는 구조로 정리하려 했다.

그런데 이 처리 방식이 DELETE 메서드의 멱등성(idempotency)에 위배되는 것 같다는 느낌을 받았다. 같은 요청을 여러 번 보냈을 때 결과가 달라지는 셈이기 때문이다.

**합의**: 존재하든 존재하지 않든 DELETE 요청에 대해서는 일괄적으로 `204 No Content`를 반환하도록 전면 수정하기로 결정했다.

### 미결 — 리뷰어 피드백을 받고 싶은 부분

#### 1. `long` vs `Long`, 왜 `Long`을 선택했는가

primitive `long`과 wrapper `Long`의 차이는 무엇인가? 우리가 `Long`을 선택한 것에 어떤 이점이 있는가? 명확한 근거를 가지고 고른 것이 아니라 짚고 넘어가고 싶다.

#### 2. DTO에서 검증을 한다면, Model에서의 검증은 여전히 필요한가

DTO에서 이름 길이 검증을 하기로 했다. 그렇다면 Model(Theme) 내부의 검증은 중복이 아닌가? 없어도 될 것 같지만 확신이 없어 일단 남겨두기로 했다.

#### 3. `Theme`과 `ThemeResponse`가 완전히 동일하다.

이럴 때 ThemeResponse DTO가 굳이 필요한가? 일단 남겨두었는데, 레이어드 아키텍처의 계층 간 구조에서 서비스가 상위 계층인 Controller에게 DTO로 반환하는 것이 상위 계층의 요구사항에 맞게 데이터를 제공하는 과정이라고 생각했다.
