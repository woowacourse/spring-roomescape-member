## 단계 3 페어 프로그래밍 회고

### 페어와 합의한 부분

#### 1. Reservation 테이블의 date 컬럼 타입 — VARCHAR 유지

Reservation 테이블의 date 속성 타입이 기존에는 VARCHAR였다. 그 이유는 제공해줬던 schema가 그렇게 되어 있어서 그냥 사용했었다.

3단계의 기능 요구사항에서 최근 일주일간의 인기 테마를 조회하려면, 오늘 날짜 기준으로 7일 이전과 오늘 날짜 사이를 표현하는 BETWEEN을 사용해야 할 것 같았다. BETWEEN은 DATE나 TIMESTAMP 같은 날짜 관련 데이터 타입에서만 유효할 것 같다고 생각해서, 스키마 타입을 VARCHAR에서 DATE로 변경해야 하지 않을까 고민했었다.

기존 VARCHAR 상태에서 BETWEEN으로 가능한지 실험을 해봤는데, 문제가 없었다. 그 이유는 날짜가 사전순 정렬이 이미 되어 있었기 때문에 BETWEEN이나 부등호로도 특정 기간을 조회할 수 있었다.

**합의**: 페어와 date 타입을 VARCHAR로 두고 남은 API를 구현하기로 결정했다.

#### 2. 정렬 기준

- Service에서 상위 10개를 많이 예약된 순으로 정렬한다.
    - 이 10개라는 정보는 Service에서 상수로 관리한다.
    - 추후 10개가 고정이 아니라면 RequestParam 같은 걸로 처리도 가능할 듯하다.
- SQL에서는 테마의 예약 개수를 우선으로 정렬한다.
    - 만약 예약된 테마 개수가 같다면, 테마의 ID 순으로 오름차순 정렬하는 것으로 합의했다.
    - 이유: 예상되는 결과가 중요하다고 판단했다.
- 만약 결과가 10개 이하면, 이하인 대로 결과를 응답한다.
    - 꼭 10개를 채워서 줘야 할까?
    - 지금 요구사항에서는 10개를 꽉 채워서 주지 않아도 될 것 같아서, 페어와 지금 구현으로 합의했다.

#### 3. 관리자와 사용자 API 분리

관리자와 사용자 API가 현재 구분이 되어 있지 않다고 느낌을 받았다.

미션 규칙에서 다음과 같은 규칙을 작성했었다.

```
3. 관리자/사용자 API 분리 기준
 (If-Then) 만약 같은 자원(리소스)을 다루더라도 행위가 같고, 목적이 다르다면 분리한다. 행위가 같고 목적도 같다면 분리하지 않는다.
→ 이유: 잘못 사용하는 것을 방지할 수 있다.
→ 이유: 목적이 같아도 나중에 관리자와 사용자 인증을 도입하여 확장하면 서로 행위가 달라질 수 있다. 그러므로 분리 필요함.
```

다시 확인해보니, 작성한 API들은 사용자와 관리자의 행동도 같고 목적도 같다고 느꼈다. 그래서 구현 과정에서 분리의 필요성을 느끼지 못했다.

**합의**: 요구사항에 보안과 관련된 얘기가 없었기에(로그인도 없음) 페어와 이 부분은 신경쓰지 않기로 합의했다.

#### 4. 테스트 레이어별 어노테이션 선택

**RepositoryTest에서는 `@JdbcTest` 활용**

- `@SpringBootTest`를 사용하면 굳이 활용하지 않는 빈들도 컨테이너에 등록되는데, 이것이 필요 없다고 판단했고, JDBC 관련 빈(JdbcTemplate 같은)만 로드하므로 성능 면에서 좋을 것이라고 판단하여 활용했다.

**ControllerTest, ServiceTest에서는 `@SpringBootTest` 활용**

- 전체 흐름을 검증하기 위해 모든 빈이 필요할 것으로 판단했다.

#### 5. 중복 시간 조회 방식

TimeRepository, ReservationRepository에서 이미 있는 시간에 대한 조회를 하기 위해 int를 활용하기로 합의했다.

- Boolean으로 해도 될 것 같긴 한데, 불필요한 조건분기가 생길 것이라 판단하여 RowMapper의 반환값을 그대로 활용하기로 했다.

### 미결 — 리뷰어 피드백을 받고 싶은 부분

#### 1. Repository 테스트에서 간단한 CRUD를 테스트해야 하는가

Repository Test에서 간단한 CRUD는 테스트하지 않기로 합의했다.

- 이유는 Repository를 테스트하는 것이 아닌, JdbcTemplate을 테스트하는 것 같은 느낌을 받았고, 다른 복잡한 SQL 쿼리를 테스트해야 한다고 판단했기 때문이다.
- 하지만 다시 생각해보니, rowMapper가 잘 동작하는지, sql문도 잘 작성이 되었는지를 확인하려면 결국에는 간단한 CRUD라도 테스트를 해야 할 것 같다고 생각하여 진행했다.
