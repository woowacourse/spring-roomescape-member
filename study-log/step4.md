## 사이클 2 솔로 프로그래밍 회고

### 고민했던 부분

#### 1. DTO 검증 응답과 비즈니스 검증 응답을 통일하려다 생긴 복잡함

Bean Validation(`@NotBlank`, `@NotNull` 등)의 에러 응답과 비즈니스 검증(`RoomescapeException`)의 에러 응답을 같은 `ErrorResponse` 포맷으로 통일하려고 했다.

그러다 보니 `@NotBlank(message = "RESERVATION_NAME_BLANK")`처럼 `message` 속성을 ErrorCode 열거형 상수 이름을 전달하는 채널로 쓰는 어색한 구조가 생겼다.

- `message` 속성은 원래 사람이 읽을 수 있는 메시지를 담는 용도인데, 열거형 이름을 넣으면 의미가 오염된다.
- `GlobalExceptionHandler`에서 `ErrorCode.valueOf(defaultMessage())`를 호출하는 순간, DTO 어노테이션 `message` 값과 `ErrorCode` 상수 이름이 반드시 일치해야 한다는 **암묵적 결합**이 생긴다. 오타나 리네임 시 런타임 에러가 발생한다.

**근본 원인**: Bean Validation(프레임워크가 처리하는 형식 검증)과 RoomescapeException(도메인이 판단하는 비즈니스 검증)은 성격이 달라서 같은 포맷으로 억지로 맞추려 하면 어딘가에서 어색함이 생긴다.

**대안으로 생각해 본 것**:
- DTO 검증은 메시지만 반환하고, 비즈니스 검증만 `ErrorCode` 기반 구조화된 응답을 반환하도록 분리
- 커스텀 어노테이션을 만들어 `code` 속성을 명시적으로 선언 (`@ErrorCodeNotBlank(code = ErrorCode.RESERVATION_NAME_BLANK)`)
---

#### 2. ErrorCode를 클라이언트에 전달하는 이유와 클라이언트의 책임

`ErrorCode`의 `message` 필드는 API 소비자(프론트엔드 개발자)에게 전달하는 메시지이고, 실제 사용자에게 보여주는 문구는 클라이언트가 직접 결정해야 한다고 생각했다.

- 서버 `message`를 그대로 화면에 보여주면 서버 구현에 종속된다. 서버가 문구를 바꾸면 화면도 바뀐다.
- `code`를 클라이언트에 전달하면, 클라이언트가 `err.code`를 보고 자신만의 사용자 친화적 메시지를 작성할 수 있다.

이 원칙에 따라 프론트엔드에서 `err.message`를 직접 노출하지 않고, `ERROR_MESSAGES[err.code]` 방식으로 클라이언트가 직접 메시지를 정의하도록 수정했다.

---

#### 3. 오늘 날짜의 지난 시간 슬롯 필터링 — 서버 vs 클라이언트

오늘 날짜를 선택했을 때 현재 시각보다 이른 시간 슬롯도 모두 표시되는 문제가 있었다.

- **클라이언트에서 필터링**: 클라이언트 시계가 서버 시계와 다를 수 있어 불일치 가능성이 있다. 또한 클라이언트가 안 보여줘도 서버는 여전히 해당 시간을 "사용 가능"으로 취급하는 불일치가 생긴다.
- **서버에서 필터링**: "과거 시간 예약 불가"는 비즈니스 룰이므로, 시간 조회 시에도 서버가 책임져야 한다. 이미 예약 생성 시 과거 날짜/시간을 막고 있으므로, 조회 단계에서도 일관성 있게 막는 것이 맞다.

**결론**: 서버에서 처리하는 것이 맞다.
