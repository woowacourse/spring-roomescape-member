# 기능 요구사항

## 1. 테마(Theme)

- [x] 테마 이름은 null이거나 빈값일 수 없다.
- [x] 테마 이름은 1자 이상 20자 이하여야 한다.
- [x] 테마 설명은 null이거나 빈값일 수 없다.
- [x] 테마 설명은 1자 이상 1000자 이하여야 한다.
- [x] 테마 URL은 null이거나 빈값일 수 없다.

### 테마 추가
- [x] 관리자는 테마를 추가할 수 있다.
- [x] 이미 존재하는 테마는 중복으로 추가할 수 없다.

### 테마 삭제
- [x] 관리자는 테마를 삭제할 수 있다.
- [x] 존재하지 않는 테마는 삭제할 수 없다.
- [x] 예약이 존재하는 테마는 삭제할 수 없다.

### 인기 테마 조회
- [x] 최근 1주 동안 예약이 많았던 테마 상위 10개를 조회한다.

---

## 2. 예약 시간(Time)

- [x] 예약 시작 시간은 null일 수 없다.
- [x] 예약 시작 시간의 형식은 "18:00"와 같은 형식을 따른다.

### 예약 시간 추가
- [x] 관리자는 새로운 예약 시간을 추가할 수 있다.
- [x] 동일한 예약 시작 시간은 중복될 수 없다.
- [x] 예약 시작 시간은 매 시간 정각이고, 소요시간은 모두 60분이다.

### 예약 시간 삭제
- [x] 관리자는 예약 시간을 삭제할 수 있다.
- [x] 존재하지 않는 예약 시간은 삭제할 수 없다.
- [x] 예약이 존재하는 예약 시간은 삭제할 수 없다.

### 사용자 예약 가능 시간 조회
- [x] 사용자는 날짜와 테마를 선택하여 예약 가능한 시간 목록을 확인할 수 있다.
- [x] 예약 가능한 시간이란, 관리자가 등록한 시간 중 해당 날짜·테마에 아직 예약되지 않은 시간이다.
- [x] 같은 날짜/시간이더라도 테마가 다르면 각각 예약 가능하다.

---

# 3. 예약(Reservation)

- [x] 예약자 이름은 null이거나 빈값일 수 없다.
- [x] 예약자 이름은 2자 이상 20자 이하여야 한다.
- [x] 예약 날짜는 null이거나 빈값일 수 없다.
- [x] 예약 날짜의 표현 형식은 "2026-05-11"와 같이 표현되어야 한다.
- [x] 예약 시간은 null이거나 빈값일 수 없다.
- [x] 예약 테마는 null이거나 빈값일 수 없다.

### 예약 생성
- [x] 예약을 생성할 수 있다.
- [x] 동일한 날짜·시간·테마 조합의 예약은 중복될 수 없다.
- [x] 존재하지 않는 예약 시간으로 예약할 수 없다.
- [x] 존재하지 않는 테마로 예약할 수 없다.
- [x] 이미 지난 날짜·시간으로는 예약할 수 없다.

### 예약 조회
- [x] 관리자는 전체 예약 목록을 조회할 수 있다.
- [ ] 사용자는 자신의 이름으로 본인의 예약 목록을 조회할 수 있다.

### 예약 변경
- [ ] 존재하지 않는 예약은 변경할 수 없다.
- [ ] 사용자는 자신의 예약 날짜와 시간을 변경할 수 있다.
- [ ] 이미 지난 예약은 변경할 수 없다.
- [ ] 변경하려는 날짜·시간에 이미 예약이 존재하면 변경할 수 없다.

### 예약 취소
- [x] 존재하지 않는 예약은 취소할 수 없다.
- [ ] 사용자는 자신의 예약만 취소할 수 있다.
- [ ] 이미 지난 예약은 취소할 수 없다.

## 4. 에러 응답 설계

- [ ] 서비스 정책 위반, 유효하지 않은 입력, 존재하지 않는 리소스 등에 대해 의도된 에러 응답을 반환한다.
- [ ] 500(서버 에러)이 사용자에게 노출되지 않도록 한다.
- [ ] 에러 응답 본문에 어떤 정보를 담을지 결정한다.
- [ ] 브라우저에서 에러 발생 시 사용자에게 의미 있는 메시지가 표시되어야 한다.

### 응답 바디 공통 형식

```json
{
  "code": "RESERVATION_DUPLICATED",
  "path": "/reservations",
  "message": "이미 존재하는 예약입니다.",
  "action": "다른 테마, 날짜, 시간으로 예약을 시도해주세요."
}
```

### 토론 규칙 문서

---

### 1. 리소스 식별 기준

- (If-Then) 만약 URL을 설계할 때
  - → [경로에 넣을 것]: 리소스(필요한 데이터) 최소 단위
  - → [쿼리스트링에 넣을 것]: 필요한 리소스를 보기 위한 필터 수단
  - → 가독성을 위해 하이픈('-')을 활용한다.
  - → 이유: 작성한 좋은 API에 부합하는 규칙이기 때문

#### (우선순위)
- URL을 결정할 때 순서:
  1. 사용자의 시나리오(유스케이스와 예상 화면)를 적어본다.
  2. 리소스(명사, HTTP 메서드)를 식별해본다.
  3. POST 메서드인 경우 body에 넣을 데이터를 설계한다.
  4. GET 메서드인 경우 쿼리스트링에 필요한 key를 설계한다.

---

### 2. 서버/클라이언트 책임 기준

- (If-Then) 만약 클라이언트가 자체적으로 판단할 수 없는 비즈니스 로직이라면
  - → 클라이언트가 자체적으로 판단할 수 없도록 서버에서 모두 가공을 해서 제공한다.
  - → 이유: 클라이언트에서 전달하는 데이터를 믿지 말자, 서버에서 데이터의 검증, 비즈니스 로직을 모두 수행하여 클라이언트에게 전달해야하기 때문.

---

### 3. 관리자/사용자 API 분리 기준

- (If-Then) 만약 같은 자원(리소스)을 다루더라도 행위가 같고, 목적이 다르다면 분리한다.
  - → 행위가 같고 목적도 같다면 분리하지 않는다.
  - → 이유: 잘못 사용하는 것을 방지할 수 있다.
  - → 이유: 목적이 같아도 나중에 관리자와 사용자 인증을 도입하여 확장하면 서로 행위가 달라질 수 있다. 그러므로 분리 필요함.


---

### 4. 좋은 API 정의

- (If-Then) 만약 API가 좋은 API라고 부르려면 URI를 확인했을 때 리소스 구조 추측이 가능하면 좋은 API이다.
  - → 이유: 가독성이 좋은 URI가 활용하기 쉽다.

- (If-Then) 클라이언트가 에러 상황에 대해서 잘 알 수 있는 API라면 좋은 API이다.
  - → 만약 에러가 발생한다면 어떤 값이 문제이고, 왜 문제인지 응답에 담아서 클라이언트에게 제공해준다.
  - → 이유: 백엔드가 프론트엔드에게 상세한 에러 상황을 줄 수 있고, 프론트가 클라이언트에게 에러 상황을 알기 쉽게 알려줘야한다.

---

### 5. 계층 별 메서드 시그니처 통일

- (If-Then) 계층 별 메서드 시그니처를 작성할 때, 일관적이게 작성한다.
  - → 이유: 가독성이 메서드가 활용하기 쉽다.

---

### 6. 비즈니스 규칙 위반 시, 요청 형식 예외 시 4XX 에러를 응답한다. (규칙 추가)

- (If-Then) 서버에서 클라이언트로 에러 메시지를 전달할 때, 다음과 같은 형태를 사용한다.
- {
  "code": "RESERVATION_DUPLICATED",
  "path": "/reservations",
  "message": "이미 존재하는 예약입니다.",
  "action": "다른 테마, 날짜, 시간으로 예약을 시도해주세요."
  }
- → 이유:
  1. code: 프론트에서 예외를 쉽게 분기할 수 있게 하기 위해서 제공합니다.
  2. path: 예외가 발생한 URI
  3. message: 프론트랑 클라이언트에게 자연어로 전달할 상세한 설명
  4. action: 그래서 이 에러를 해결하기 위해 프론트 혹은 클라이언트가 무엇을 해야할지에 대한 설명 (nullable)

### 금지 사항

- (금지) 이번 사이클에서 명확한 이유가 없다면 도입하지 하지 않는다
  - → 이유: 틀려도 괜찮다. 다만 명확한 이유를 가지고 코드 작성을 해보는 습관을 기르고자 한다.

- (금지) URL에 http method 붙이지 않는다.
  - (예: /get/books, /books/1/delete)
  - <https://www.youtube.com/watch?v=RP_f5dMoHFc> 참조
  - → 언더바(_)는 쓰지 않는다.
