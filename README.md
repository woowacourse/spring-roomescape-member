## API 명세

### 관리자 API (`/admin/...`)

| 기능    | Method / URL                      | 요청 본문                           | 응답                                     |
|-------|-----------------------------------|---------------------------------|----------------------------------------|
| 예약 조회 | `GET /admin/reservations`         | -                               | `[{id, name, date, time, theme}, ...]` |
| 예약 추가 | `POST /admin/reservations`        | `{name, date, timeId, themeId}` | `{id, name, date, time, theme}`        |
| 예약 삭제 | `DELETE /admin/reservations/{id}` | -                               |                                        |

#### 시간 관리

| 기능    | Method / URL               | 요청 본문       | 응답                     |
|-------|----------------------------|-------------|------------------------|
| 시간 조회 | `GET /admin/times`         | -           | `[{id, startAt}, ...]` |
| 시간 추가 | `POST /admin/times`        | `{startAt}` | `{id, startAt}`        |
| 시간 삭제 | `DELETE /admin/times/{id}` | -           |                        |

#### 테마 관리

| 기능    | Method / URL                | 요청 본문                            | 응답                                          |
|-------|-----------------------------|----------------------------------|---------------------------------------------|
| 테마 조회 | `GET /admin/themes`         | -                                | `[{id, name, description, thumbnail}, ...]` |
| 테마 추가 | `POST /admin/themes`        | `{name, description, thumbnail}` | `{id, name, description, thumbnail}`        |

### 사용자 API (`/user/...`)

> **사용자 흐름:** 테마를 본다 → 날짜를 고른다 → 그 조건에서 예약 가능한 시간을 본다 → 시간을 골라 예약한다.

| 기능          | Method / URL                                                 | 요청 본문                           | 응답                                                            |
|-------------|--------------------------------------------------------------|---------------------------------|---------------------------------------------------------------|
| 테마 목록 조회    | `GET /user/themes`                                           | -                               | `[{id, name, description, thumbnail}, ...]`                   |
| 예약 가능 시간 조회 | `GET /user/themes/{themeId}/available-times?date=YYYY-MM-DD` | -                               | `[{id, startAt}, ...]`                                        |
| 사용자 예약 추가   | `POST /user/reservations`                                    | `{name, date, timeId, themeId}` | `{id, name, date, time, theme}`                               |
| 인기 테마 조회    | `GET /user/themes/popular`                                   | -                               | `[{id, name, description, thumbnail, reservationCount}, ...]` |

## 좋은 API를 설계하기 위한 4가지 기준

### [1] 리소스 식별 기준

URL을 정할 때 무엇을 리소스로 보고, 무엇을 필터/표현 조건으로 보는가.   
**같은 데이터를 두 개 이상의 URL로 표현할 수 있다면 → 사용자 흐름이 더 잘 드러나는 쪽을 선택한다.**

같은 쿼리 결과라도 "이 도메인을 어떻게 바라보는가"가 URL의 의미를 결정한다.   
예약 가능 시간은 `/times?themeId=&date=`로도 표현 가능하지만,   
"사용자가 테마를 고르고 → 날짜를 고르고 → 가능 시간을 본다"는 흐름을 따라   
`/themes/{id}/available-times?date=`를 선택했다. URL은 도메인 의도를 보존하는 1차 문서다.

**URL을 결정할 때 우선순위**

1. **무엇을 리소스(명사)로 볼 것인가**부터 정한다.
    - 도메인에서 명사로 부를 수 있는 것과 조건/표현으로 부를 것을 가른다.
2. **행위는 HTTP 메서드로 표현한다.**
    - 동사형 URL(`/getPopularThemes`, `/createReservation`)을 쓰지 않는다.
3. **부가 조건은 의미에 맞게 배치한다.**
    - 리소스를 식별하는 값은 path, 필터/정렬은 query parameter, 새로 만들 데이터는 body.

1번이 흔들리면 2,3번이 바뀌어도 일관성이 무너진다. "예약 가능 시간"의 결정이 1번에서 갈렸다.

---

### [2] 서버/클라이언트 책임 기준

어떤 판단을 서버가 책임지고, 어떤 판단을 클라이언트가 책임지는가.    
**클라이언트가 가진 정보만으로는 판단할 수 없는 조건이라면 → 서버가 책임지고 응답에서 걸러서 내려준다.**

예약 가능 여부는 전체 사용자의 예약 정보를 알아야 판단할 수 있는데, 그 정보의 주체는 서버뿐이다.   
클라이언트에게 모든 예약 데이터를 내려주고 계산하게 만드는 것은 데이터 노출과 책임 소재 양쪽에서 부적절하다.   
URL 이름이 `available-times`라면 "예약 가능한 것만 내려준다"는 의미가 자연스럽게 따라온다.

**같은 데이터를 `서버 필터 결과만` vs `전체 + 상태 플래그`로 내릴지 고민된다면 → 응답이 표현하려는 의미가 무엇인지로 결정한다.**

두 방식은 동작은 같지만 의미가 다르다.    
"예약 가능한 시간 목록"을 표현하려면 가능한 것만 내리는 게 자연스럽고,   
"전체 시간표 + 마감 표시"를 표현하려면 플래그 방식이 자연스럽다.   
이번 미션에서는 가이드 문장이 전자에 가까워 서버 필터를 선택했다.     
응답 형태는 "어떻게 가공할까"가 아니라 "무엇을 표현하는가"의 문제다.

---

### [3] 관리자/사용자 API 분리 기준

같은 자원을 다루는 관리자와 사용자 API를 분리하는가, 합치는가.   
**같은 행위라도 주체/목적/검증 규칙이 다르거나, 가까운 미래에 달라질 것이 예상된다면   
→ URL을 분리한다 (`/admin/...`, `/user/...`).**

같은 "예약 생성"이라도 관리자는 "전화/현장 예약 등록"이고, 사용자는 "예약 가능 시간 내에서 직접 확정"이다.   
검증 규칙이 다르고, 응답에 필요한 정보의 양도 다를 수 있다.   
현재의 차이뿐 아니라 예상되는 차이까지 고려해서 분리하는 것이 옳다고 봤다.

---

### [4] "좋은 API"의 정의

동작하는 것을 넘어, 어떤 조건을 만족해야 좋은 API라고 부를 것인가.
> **좋은 API는 URL/메서드/응답만 보고도 "왜 이렇게 설계했는가"를 추론할 수 있는 API다.**

**좋은 API의 조건**

1. **의도의 가시성** - 동사형 URL이 아니라 명사형 리소스로 의도가 드러난다. (`/themes/popular`는 별도 유스케이스임을 URL이 말해준다.)
2. **일관성** - 시스템 전체에서 같은 컨벤션이 깨지지 않는다. 깨야 한다면 "왜 깨야 했는가"가 설명 가능해야 한다.
3. **책임 경계의 명시성**
    - 응답 형태가 "이 판단의 책임이 어디에 있는가"를 드러낸다.
    - 서버 필터 결과만 내리는 것은 단순한 형식이 아니라 책임의 선언이다.

**안 좋은 API의 징후**

- 동사형 URL (`/getPopularThemes`, `/reservations/create`)
- 같은 리소스인데 prefix 컨벤션이 일관되지 않음 (이유 없이)
- 응답이 너무 많이 또는 너무 적게 가공됨
- 무엇을 표현하려 했는지 응답에서 안 읽힘
- "왜 이 메서드/상태 코드를 골랐는가"에 "관례라서" 외에 답이 없음

동작은 시작점일 뿐, 좋은 API는 다른 개발자가 6개월 뒤에 봐도 설계자의 판단을 추론할 수 있는 API라고 생각한다.
