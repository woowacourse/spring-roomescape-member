## 2. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?

  락, Undo Log는 바로 떠올렸지만 Dirty Checking은 기억이 안 났다 — 연결이 덜 된 부분을 직접 발견했다.

### 학습 과정

#### `readOnly = true`의 효과 3가지

**DB 레벨**

- **락 생략** — 읽기 전용임을 알기 때문에 불필요한 락을 걸지 않는다
- **Undo Log 생략** — 데이터를 수정하지 않으므로 이전 버전을 기록할 필요가 없다

**JPA 레벨**

- **Dirty Checking 스냅샷 생략** — JPA는 엔티티 변경 감지를 위해 스냅샷을 찍는데, `readOnly = true`면 이 스냅샷 자체를 안 찍는다 → 메모리와 연산 비용 절감

**트랜잭션 레벨**

- **일관된 스냅샷 보장** — 트랜잭션 없이 여러 번 조회하면 매번 다른 스냅샷을 볼 수 있다. `readOnly = true`로 트랜잭션을 열면 그 안에서는 일관된 스냅샷을 보장받는다

#### 실무 패턴

```java

@Service
@Transactional(readOnly = true)  // 클래스 레벨 — 조회가 대부분
public class ReservationService {

    @Transactional  // 메서드 레벨 오버라이드 — 수정이 필요한 경우만
    public Reservation create(...) { ...}
}
```

조회가 대부분인 서비스에서 클래스 레벨에 `readOnly = true`를 걸고, 수정 메서드만 `@Transactional`로 오버라이드하는 게 표준 패턴이다.

## 3. 전략 평가

- 효과적이었던 것과 그 이유
  MVCC → Undo Log → readOnly 순으로 자연스럽게 연결됐다.
  이전 사이클에서 배운 개념이 새 개념의 이해를 도운 나선형 학습의 효과가 느껴졌다.

- 비효과적이었던 것과 그 이유
  인출 연습에서 Dirty Checking이 바로 안 나왔다. JPA 레벨 효과는 DB 레벨보다 연결고리가 약하다 — 다음 사이클에서 영속성 컨텍스트와 다시 연결해볼 필요가 있다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  이월 키워드를 연결 질문 방식으로 풀어나간 게 잘 맞았다.

## 4. AI 피드백

인출 연습에서 Dirty Checking을 놓쳤는데, 이게 중요한 신호다. 락과 Undo Log는 DB 관점으로 직관적이지만, Dirty Checking은 JPA 내부 동작 관점이라 연결이 약하다. 다음 사이클에서
영속성 컨텍스트를 다시 볼 때 이 연결을 의도적으로 만들어두면 세 가지가 하나로 묶인다.
