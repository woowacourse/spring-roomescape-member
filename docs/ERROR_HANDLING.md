# 오류 처리 가이드 (Error Handling Guide)

이 문서는 방탈출 예약 시스템 API에서 발생하는 오류 응답의 구조와 정책을 설명합니다.

## 1. 개요

본 프로젝트는 클라이언트와의 통신 일관성을 위해 특정 예외 상황에서 일반적인 HTTP 상태 코드 규약과는 다른 정책을 사용하고 있습니다. 모든 에러 메시지는 기본적으로 `text/plain;charset=UTF-8` 형식으로 반환됩니다.

## 2. 특별 정책: NotFoundException (200 OK)

시스템 내에서 리소스를 찾을 수 없는 경우 발생하는 `NotFoundException`은 HTTP 상태 코드 **`200 OK`**를 반환합니다.

### 2.1. 정책 배경
- **오류 성격의 명확한 분리**: 존재하지 않는 API 경로(URL)로 잘못 요청한 경우(인프라/라우팅 오류, 404 Not Found)와, API 경로는 유효하나 요청한 조건의 데이터가 존재하지 않는 상황(비즈니스 예외, `NotFoundException`)을 명확히 구분하기 위함입니다.
- **클라이언트 측 처리 편의성**: 4xx 에러를 시스템이나 네트워크 레벨의 중대한 오류로 간주하여 일괄 처리하는 클라이언트 환경에서, '데이터 없음'과 같은 비즈니스적 예외 상황을 정상 흐름(200 OK) 내에서 유연하게 처리할 수 있도록 합니다.

### 2.2. 대상 사례
- 존재하지 않는 예약 ID로 조회/수정/삭제 요청 시
- 존재하지 않는 테마 ID 또는 시간 ID로 요청 시
- 본인의 예약이 아닌 경우 (이름 불일치로 인해 해당 예약을 '찾을 수 없음'으로 간주)

---

## 3. HTTP 상태 코드별 응답

| 상태 코드 | 의미 | 설명 |
| :--- | :--- | :--- |
| **200 OK** | 성공 / 데이터 없음 | 요청이 성공했거나, `NotFoundException`이 발생한 경우입니다. |
| **400 Bad Request** | 잘못된 요청 | 필수 파라미터 누락, 형식 오류 등 요청 자체가 잘못된 경우입니다. |
| **403 Forbidden** | 권한 없음 | 관리자 전용 API에 유효한 권한(Header) 없이 접근한 경우입니다. |
| **409 Conflict** | 비즈니스 규칙 위반 | 중복 예약, 사용 중인 테마/시간 삭제 시도 등 데이터 충돌 발생 시 사용합니다. |
| **422 Unprocessable Entity** | 유효하지 않은 데이터 | 과거 날짜로 예약 시도 등 데이터의 논리적 유효성이 떨어지는 경우입니다. |
| **500 Internal Server Error** | 서버 오류 | 서버 내부에서 예상치 못한 에러가 발생한 경우입니다. |

---

## 4. 공통 에러 메시지 (ErrorCode)

주요 에러 메시지는 아래와 같으며, 응답 본문(Body)에 텍스트 형식으로 포함됩니다.

### 예약 관련
- `예약을 찾을 수 없습니다. 예약 ID를 확인해주세요.` (200)
- `예약을 찾을 수 없습니다. 예약자 성함을 확인해주세요.` (200)
- `동일한 날짜, 시간, 테마의 예약이 이미 존재합니다. 다른 조건을 선택해주세요.` (409)
- `지나간 날짜나 시간으로는 예약을 생성하거나 삭제할 수 없습니다. 현재 이후의 시간을 선택해주세요.` (422)

### 관리자 및 권한 관련
- `관리자 권한이 필요합니다.` (403)
- `요청 이름이 비어있습니다. 이름을 입력해주세요` (400)

### 테마 및 시간 관련
- `테마를 찾을 수 없습니다. 테마 ID를 확인해주세요.` (200)
- `예약 시간을 찾을 수 없습니다. 예약 시간 ID를 확인해주세요.` (200)
- `사용 중인 예약 시간은 삭제할 수 없습니다. 관련 예약을 먼저 삭제해주세요.` (409)
- `사용 중인 테마는 삭제할 수 없습니다. 관련 예약을 먼저 삭제해주세요.` (409)
