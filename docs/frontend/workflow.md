### 1. 요구사항 분석

- 무엇을 만들어야 하는지 기능 단위로 정리한다.
- 사용자/관리자 등 액터를 구분한다.
- 핵심 정책과 제약 조건을 표시한다.

**산출물**

- 기능 목록
- 액터 목록
- 핵심 정책/제약 조건

---

### 2. 사용자 흐름 정의

- 액터를 적는다.
- 목표를 한 문장으로 적는다.
- 성공 흐름을 순서대로 적는다.
- 대체/실패 흐름을 적는다.

**주의**

- 화면, API, 구현 방식까지 들어가지 않는다.
- “사용자가 무엇을 하는가”에 집중한다.

**산출물**

- 액터
- 목표
- 성공 흐름
- 대체/실패 흐름

---

### 3. 화면 목록 정의

- 사용자 흐름을 기준으로 필요한 화면을 뽑는다.
- 각 화면의 목적을 한 문장으로 적는다.
- 화면 간 이동 흐름을 적는다.

**주의**

- 컴포넌트나 API까지 들어가지 않는다.
- 목적이 다른 화면은 분리한다.

**산출물**

- 화면 목록
- 화면별 목적
- 화면 이동 흐름

---

### 4. 화면 상태 정의

- 각 화면에서 필요한 상태를 적는다.
- 클라이언트 상태와 서버 상태를 구분한다.
- 로딩/에러/빈/성공 상태를 표시한다.

**산출물**

- 화면별 state 목록
- Client State
- Server State
- Loading/Error/Empty/Success 상태

---

### 5. API 사용 흐름 정의

- 어떤 사용자 행동에서 어떤 API가 호출되는지 연결한다.
- 요청 성공/실패 시 화면 반응을 적는다.
- 필요한 요청/응답 데이터를 정리한다.

**산출물**

- 사용자 행동 → API 호출 목록
- 성공 시 처리
- 실패 시 처리
- 필요한 요청/응답 데이터

---

### 6. 컴포넌트 설계

- 화면을 책임 단위로 나눈다.
- 재사용 가능한 UI를 분리한다.
- props로 전달할 값을 정리한다.

**산출물**

- 컴포넌트 트리
- 컴포넌트별 책임
- 주요 props

---

### 7. Custom Hook 설계

- 컴포넌트에서 상태/로직을 분리한다.
- API 호출, 폼 상태, 검증 로직을 hook으로 분리할지 판단한다.

**산출물**

- Custom Hook 목록
- Hook별 책임
- 입력값/반환값

---

### 8. UI 상태 테스트

- 사용자 행동 기준으로 테스트 시나리오를 작성한다.
- 로딩/에러/빈/성공 상태를 검증한다.
- API 성공/실패 케이스를 검증한다.

**산출물**

- 사용자 시나리오 테스트 목록
- 상태별 테스트 목록
- API 성공/실패 테스트 목록

---

### 9. 구현

- 화면 뼈대부터 만든다.
- API를 연결한다.
- 상태 처리를 완성한다.
- 컴포넌트와 hook을 리팩터링한다.

**산출물**

- 동작하는 화면
- API 연결
- 테스트 통과
- 리팩터링 완료
