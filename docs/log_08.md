### 학습 과정

#### JPA vs JDBC 낙관적 락

`@Version`은 JPA `@Entity`에서만 동작한다. 이 프로젝트는 JDBC 기반이라 직접 구현해야 한다.

#### JDBC 낙관적 락 구현 원리

`@Version`이 내부적으로 하는 일을 SQL로 직접 구현한다.

```sql
UPDATE reservations
SET ..., version = version + 1
WHERE id = ? AND version = ?
```

- 버전이 맞으면 → UPDATE 성공, `version + 1`로 갱신
- 버전이 다르면 → 영향받은 행 0 → `ConflictException` 발생 → 클라이언트 409 응답

#### 구현 내용

1. **schema.sql** — `version BIGINT NOT NULL DEFAULT 0` 추가
2. **Reservation** — `version` 필드 추가
3. **ReservationJdbcDao.update()** — `WHERE id = ? AND version = ?` 조건 추가, 반환값 0이면 예외

```java
int updated = jdbcTemplate.update(sql, params);
if (updated == 0) {
    throw new ConflictException("다른 사용자가 이미 수정했습니다. 다시 시도해주세요.");
}
```

#### retry vs 예외 반환

retry는 서버에서 자동으로 재시도하는 방식이고, 이번 구현은 클라이언트에게 409를 반환해서 클라이언트가 다시 요청하도록 했다. 서버 retry는 별도 구현이 필요하다.

## 3. 전략 평가

- 효과적이었던 것과 그 이유
  log_06에서 개념을 이해했기 때문에 구현 방향이 바로 잡혔다. "버전이 맞지 않으면 반환값이 0"이라는 추론을 스스로 했고, 이게 구현의 핵심이었다.

- 비효과적이었던 것과 그 이유
  retry 구현까지는 다루지 않았다. 실제 서비스라면 서버 레벨 retry 전략(`@Retryable` 등)도 고려해야 한다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  개념 → 코드 적용 순서로 진행한 게 잘 맞았다. 나선형 학습에서 사이클이 제대로 닫힌 느낌이다.
