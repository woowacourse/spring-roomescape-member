# Mocking을 왜 해야할까?

mocking은 외부 자원이나 외부에서 영향을 주는 요소가 있을 때 그 요소에 영향을 받지 않고 테스트를 하기 위해 필요하다. 대표적으로 현재 시간, DB/파일 액세스, 외부 API 결과 등이 있다.

## 기타 테스트 관련 단어

1. mock
    * 행위를 검증하여 테스트하는 방법. 어떤 메소드가 호출되었는지, 또는 몇 번 호출되었는지를 테스트한다.
    * 서비스의 repository를 테스트할 때 사용
2. stub
    * 상태 검증에 활용한다. `when().thenReturn()`으로 이 메서드가 이 상태의 객체를 반환해야 함을 가정해둔다.
    * 서비스의 repository를 테스트할 때 사용
3. spy
    * 실제 객체에서 일부 기능만 가짜로 동작하게 바꾸고 호출 테스트할 때 사용한다.
    * 언제 쓰면 좋을지는 불확실
4. dummy
    * 아무 기능도하지 않는 객체
    * 테스트 시 필요한 객체지만 검증 대상이 아닐 때 사용할 수 있다.
    * 서비스 테스트에서 테스트 대상이 아닌 policy가 필요할 때 임시로 사용했다.
5. fakes
    * 실제로 사용되지는 않지만 같은 동작을 하게 구현된 객체
    * 임시로 `List<>`에 든 값에서 그 결괏값을 인터페이스의 메소드가 요구하는 대로 뽑아내는 객체가 해당된다

## 정리

예전에는 mocking이 외부 요소가 테스트에 영향 주지 않도록 분리한 객체라 생각했다. 이번 리뷰를 통해 test doubles에 대해 찾아보고 행위 검증과 상태 검증(상태 가정에 가까운거 같긴하다)에 
필요한 방법이라고 알게됐다. mockito가 두 방법 다 지원하는 것일 뿐임을 알았다. 지금은 외부 요인과 분리를 하기 위한 수단이 stub과 mock 둘 다 인걸로 본다.
