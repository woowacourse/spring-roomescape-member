### 학습법 v1: 목표 기반 + 필사 중심

**핵심 전략**: "학습이 끝나면 할 수 있어야 하는 것 3가지"를 설정하고, 학습 자료를 노션에 필사하며 공식 문서를 통독하는 방식.

**잘 동작한 점**
- "할 수 있어야 하는 것 3가지" 설정이 학습 방향을 잡는 데 효과적이었다. 학습 중간과 종료 시점에서 초기 목표대로 진행했는지 확인할 수 있었다.

**충돌 지점**
- 학습 자료를 그대로 노션에 따라 쓰는 필사는 이해 모드가 아닌 복사 모드로 전환되어 시간 대비 학습 효과가 낮았다.
- 공식 문서를 처음부터 끝까지 통독하는 것은 레퍼런스 학습이지 목표 지향 학습이 아니었다.

### 학습법 v2: 점진적 과부하 + 스파이럴 학습 도입

**핵심 전략**: 익숙한 스택에서 요소 하나만 낯선 것으로 교체해 난이도를 올리고, 학습 과정을 기록하며 진행한다. 잘못된 방향으로 왔다고 인지한 시점에서 체크포인트로 돌아와 문제를 재정의하고 새 방향으로 학습을 반복한다.

**잘 동작한 점**
- SOLID 원칙 학습 시, 꼬리를 무는 궁금증(리스코프 치환 원칙의 이름 유래 등)이 생겼지만 기록을 남기며 원래 목표로 돌아올 수 있었다.
- AI를 활용해 설계 판단은 직접 내리고 구현 보조는 AI에게 맡기는 분리가 자리 잡았다.
- 필사 대신 빠르게 읽고 테스트 코드를 풀며 지식을 습득하는 방식으로 전환했다.

**충돌 지점**
1. **깊이 vs 범위 판단 기준 부재**: "이 방향으로 깊이 들어가는 것이 맞는가?"라는 의문이 종종 들었다. MongoDB Aggregation Root까지 파고 들었지만 미션 핵심은 DB 연동이었고, 깊이와 범위 사이에서 판단 기준이 없었다.
2. **학습 자원 혼란**: 새로운 기술 학습 시 공식 문서 → 블로그 → 영상 순으로 봤는데 각각 설명이 달라서 어디서부터 손대야 할지 모르겠었다.
3. **기록 실패**: 잘못된 방향에서 돌아올 때 상황과 배운 점을 기록하기로 했지만, 시간 압박에 생략해버렸다.

| 항목 | v2 |
|------|-----|
| 핵심 전략 | 점진적 과부하 (하나씩 교체) |
| 이해도 확인 | 코드가 돌아가면 통과 |
| 깊이 판단 | 기준 없음 (감으로 멈춤) |
| 회고 방식 | 학습 시간 내 기록 시도 → 실패 |
| 학습 자원 | 공식문서 + 블로그 + 영상 혼합 |
| 방향 감지 | 사후에 "잘못 왔다" 인지 |

### 학습법 v3: 점진적 과부하 + 5분 설명 녹음

기존 점진적 과부하 학습법에서 가장 부족했던 점은 "코드가 돌아가면 이해했다고 넘어가는 것"이었다. 이해했다는 착각을 걸러낼 거울이 없었기 때문이다. 이를 개선하기 위해 "5분 설명 녹음"을 추가했다.

**마무리한 학습법**

1. 익숙한 스택에서 요소 하나만 낯선 것으로 교체해 난이도를 올린다.
2. 학습 자원은 공식 문서와 블로그 예제 두 가지로만 제한한다.
3. 구현 전에 "학습이 끝나면 할 수 있어야 하는 것 3가지"를 적어 깊이의 경계를 정한다.
4. 학습 스텝마다 "지금 문제를 풀고 있는가?"를 자문해 방향을 잡는다.
5. 구현이 끝나면 자료를 모두 닫고 3~5분간 일상어로 설명하여 막히는 지점을 찾아 재학습한다.
6. 학습 종료 후 짧은 회고로 마무리한다.

**v2 → v3 변경 요약**

| 항목 | v2 | v3 |
|------|-----|-----|
| 핵심 전략 | 점진적 과부하 (하나씩 교체) | 점진적 과부하 + 5분 설명 녹음 |
| 이해도 확인 | 코드가 돌아가면 통과 | 설명이 막히는 지점 = 이해 못한 부분 |
| 깊이 판단 | 기준 없음 (감으로 멈춤) | "할 수 있어야 하는 것 3가지"로 경계 설정 |
| 회고 방식 | 학습 시간 내 기록 시도 → 실패 | 학습 종료 후 별도 2분 회고 |
| 학습 자원 | 공식문서 + 블로그 + 영상 혼합 | 공식문서 + 공식 예제만 (블로그는 검증 단계에서) |
| 방향 감지 | 사후에 "잘못 왔다" 인지 | 25분마다 "문제를 풀고 있는가?" 자문 |

**PiFSR 중 이번 주에 의식하고 싶은 원칙: Problem-Driven**

어떤 것이든 문제 자체를 정의하는 일이 가장 어려운 일이라고 생각한다. 미션을 시작할 때 코드부터 치지 않고 "이 미션이 해결하려는 문제가 뭔지"를 한 문장으로 먼저 적고, 진행 중에 잘못 정의했다면 다시 문제 정의로 돌아와서 재정의한다.

### 미션 진행 중 학습법의 실제 변화

v3를 설계한 시점과 실제 미션을 진행한 4개의 타임을 거치면서, 학습법이 v3와는 다른 방향으로 자연스럽게 변화했다.

v3에서 핵심으로 추가한 "5분 설명 녹음"이나 "25분마다 자문" 같은 장치보다, 미션을 반복 수행하면서 체득한 다음 습관들이 실제로 더 효과적이었다.

**실제로 자리 잡은 학습법 (log-04 가 기준이됨.)**

1. **필사하지 않고 빠르게 읽고 바로 적용한다.** 학습 자료를 생각하며 읽고, 테스트 코드를 직접 풀면서 지식을 확인한다.
2. **AI는 적극적으로 참고하되, 정답으로 여기지 않는다.** 설계 판단뿐만 아니라 AI가 추천한 코드도 현재 상황에 맞는지 직접 판단한다. (log-04에서 AI의 과도한 추상화 추천을 거부한 경험)
3. **문서 종류에 따라 전략을 분기한다.** 개념 설명형 문서는 읽고 핵심을 정리하고, 레퍼런스형 문서는 북마크만 해두고 필요할 때 찾아본다.
4. **익숙한 영역은 빠르게 끝내고 시간을 확보한다.** 불필요하게 시간을 끌지 않고, 확보한 시간으로 선택 미션 등 추가 도전을 한다.

v3를 출발점으로 삼았지만, 미션을 직접 부딪히며 만든 이 학습법이 현재 실제로 사용하는 방법이다. 설계된 학습법과 실전에서 작동하는 학습법 사이에 차이가 생긴 것은 자연스러운 일이며, 이 차이를 인식하고 있다는 것 자체가 성장이라고 생각한다.

주말에 시도할 것: 학습 자료를 열기 전에 "30초 목차 스캔 → 통독/검색 판단"을 먼저 하는 입구 루틴 만들기
적용 시나리오:

[언제·어떤 상황에서] 새로운 학습 자료(공식문서, README, 블로그 등)를 열 때마다, 코드를 치기 전에 30초 타이머를 켜고 목차·헤더만 훑는다. 그 뒤 "이 문서는 지금 처음부터 읽을 것인가, 북마크해두고 구현 중 필요할 때 검색할 것인가?"를 한 줄로 적는다. 적지 않으면 읽기를 시작하지 않는다.
[그다음 단계] 통독으로 판단했으면 → 읽되 필사하지 않고, 읽은 뒤 자료를 닫고 "모르는 것만" 질문 형태로 적는다. 검색용으로 판단했으면 → 바로 코드 구현을 시작하고, 막히는 지점이 생겼을 때만 해당 섹션을 찾아본다.
[멈출 조건] 이 루틴을 3회 연속 적용했는데도 결국 매번 "통독"을 선택하고 있다면, 판단 기준이 아니라 불안이 결정을 내리고 있다는 신호다. 그때는 루틴 자체를 바꾸는 대신, 다음 1회만 강제로 "검색용"을 선택하고 구현을 먼저 시작해본 뒤 결과를 비교한다.
