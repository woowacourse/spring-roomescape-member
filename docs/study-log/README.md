## 학습법 변천

### v1 — 레벨1 미션 사이클 학습법

레벨1 미션(블랙잭, 장기)에서 실제로 해온 7개 원칙. 미션이 출제자 의도라는 명확한 출발점을 줘서 잘 작동했음.

1. 출제자 의도 파악 — 시작점
2. AI 배제 → 본인 머리로 설계 완성
3. AI는 평가자로만, 답 금지 역질문만
4. 모르는 개념은 바텀업
5. 코드 리뷰 — 한 항목씩, 영향 범위 확인 후 다음
6. 반대 의견은 근거로 푸시백
7. 한 두 줄 리뷰도 깊게 파기 (질문 의도 한 번 더)

### v2 (2026-04-28, 토론 활동) — 자기주도 학습용 보완

레벨2 이후 자기주도 학습 비중이 늘면 1번(출제자 의도) 출발점이 사라지는 문제 인식. 그 빈자리를 채우려는 첫 시도.

- **8. 문제 정의를 AI와 함께 뾰족하게** — PiFSR의 P 강화. AI는 정의 좁히는 도구일 뿐 답을 주지 않음 (3번과 일관)
- **9. i+1 — 끊는 기준**: *이 전제를 더 파야 본 개념 막힘이 풀리는가?* 풀리지 않으면 본 개념으로 복귀
- **10. 결론 명시 → 메모이제이션** (출처: 토론 동료 서연)

### v3 (2026-04-28, gRPC 회고) — 시간 차원 추가

gRPC 자기주도 학습으로 v2 첫 검증. 두 결함 발견. AI 학습 전문가(Ericsson, Oakley) 피드백으로 보완.

- **8a. 30초 추측** (Ericsson — 의도적 연습): AI에 답 받기 전 인접 지식으로 추측 → 답 받은 뒤 어디가 맞고 틀렸는지 비교
- **11. 24시간 후 빈 종이 인출** (Oakley — Focused/Diffuse Mode): 노트 안 보고 결론 다시 쓰기. 입력보다 인출이 5~10배 효과

## 이번 미션에서의 적용 결과

본 미션은 **자기주도가 아니라 미션 사이클**이라 v1(1~7번)이 주축이 됐다. v2/v3의 8~11번은 적용 비중이 낮았다.

| 원칙 | 적용 정도 | 회고 |
|------|----------|------|
| 1. 출제자 의도 파악 | ✅ 높음 | "왜 4단계 분리를 시켰나" 의식하며 진행 |
| 2. AI 배제 (설계) | ⚠️ 부분 | 1단계는 답 위주, 4단계 리팩토링은 본인 판단 후 검증 |
| 3. AI 답 금지 | ⚠️ 부분 | 미션 처음이라 답 받음. 깊이 학습 단계에선 역질문 위주 |
| 4. 바텀업 | ✅ 강함 | RowMapper → ResultSet → JDBC → by-value 순으로 깊이 들어감 |
| 5. 한 항목씩, 영향 범위 | ✅ 강함 | 리팩토링 시 매번 영향받는 파일 확인 후 다음 |
| 6. 근거로 푸시백 | — | 리뷰어 없는 단계라 미적용 |
| 7. 깊이 파기 | ✅ 강함 | "Object 왜 쓰지?", "int 형 맞아?" 등 사소한 의문 추적 |
| 8a. 30초 추측 | ❌ | 미션 사이클엔 부담. 자기주도 학습용 |
| 9. i+1 끊는 기준 | ✅ 강함 | "더 파면 본 개념 막힘 풀리나?"로 자주 끊음 |
| 10. 결론 명시 | ✅ 강함 | 매 라운드 끝에 한 줄 결론 + 커밋 메시지 |
| 11. 24시간 인출 | ❌ | 미적용. 다음 미션에서 시도 예정 |

**핵심 발견**: v3의 8a/11번은 자기주도 학습 도구라 미션 사이클에는 적용 부담이 컸다. **미션 사이클 ≠ 자기주도 학습**이라는 모드 분리가 v4의 출발점이 된다.

## v4의 본질 — 학습 메타 에이전트

> 상세 명세는 [log-04](./log-04.md) Part 2 참조.

### 발견한 패턴

학습 로그 작성 라운드에서 v4 후보 4가지(아래 A/B/C/D)가 **별개 도구가 아니라 하나의 시스템**임을 발견했다.

4가지 모두 다음 공통점을 갖는다:

1. **학습 본문 외부**의 작업이다 (시작 전 또는 종료 후)
2. **반복적**이다 (매 학습 사이클마다 같은 작업)
3. **자동화 가능**하다 (옵시디언 데이터 + 프롬프트 템플릿)
4. **연결돼 있다** (하나의 흐름)

따라서 v4는 4가지 별개 원칙이 아니라 **학습 메타 에이전트 = 학습법의 운영 엔진**이다.

### v4 구성 요소 (= 에이전트의 4개 부품)

#### A. 키워드 1차 빠른 학습 → 계획 → 본 학습
처음부터 깊이 들어가지 말고, 아는 선에서 키워드를 1차로 훑어 지도를 그린 뒤 그걸 기반으로 학습 계획을 짜고 본 학습으로 진입.

→ **에이전트의 Pre-learning 단계**: 8a(30초 추측) 후 키워드 1차 훑기 가이드 트리거.

#### B. 옵시디언 정리 프롬프트 사전 정교화
AI에 학습 들어가기 전, 결과를 어떻게 옵시디언에 정리할지 프롬프트를 미리 정교하게 만든다. 학습 후 그 프롬프트로 정리 요청.

→ **에이전트의 Post-learning 자동 정리**: 정리 프롬프트가 자동 실행. 외화 비용을 거의 0으로 낮춤.

#### C. 옵시디언 방향그래프 + 위상정렬
학습 키워드 의존성을 옵시디언 그래프로 시각화하되 방향그래프로 둔다. 다음 학습 시작 시 위상정렬된 사전 자료 + 핵심 요약을 받고 시작.

→ **에이전트가 그래프를 자동 관리**. v2의 "학습 키워드 위상 정렬" 충돌의 직접적 해결안.

#### D. 피드백 전문가별 프롬프트 분리
v3의 AI 학습 전문가 피드백 메커니즘을, 전문가 1명당 프롬프트를 미리 분리해 저장. 그때그때 유연하게 선택.

→ **에이전트의 종료 시 옵션**: Post-learning에서 어느 전문가에게 피드백 받을지 선택 메뉴.

### 에이전트 활성화 흐름

```
[학습 시작]
   ↓ 마크 클릭
[Pre-learning] — 카테고리 선택 / 사전 자료 패키지 / P 좁히기 / 추측 트리거
   ↓ 컨텍스트 + 정리 프롬프트 전달
[학습 본문] — 클로드와 학습 (에이전트는 백그라운드)
   ↓ 종료 후 마크 재클릭
[Post-learning] — 옵시디언 자동 정리 / 그래프 업데이트 / 다음 학습 후보 / (선택) 11번 알림
```

### v2/v3 충돌 지점의 해결

| 충돌 | 에이전트의 해결 |
|------|---------------|
| 외화의 지속 비용 | 자동 정리로 비용 0에 가깝게 |
| 학습 키워드 위상 정렬 | 그래프 자동 관리 |
| AI 배제 vs 활용 결정 부담 | 단계별 모드 분리 자동화 |
| 8a 추측의 진정성 | 강제 단계로 둠 |
| 11번 24h 인출 지속 가능성 | 알림 자동화 |

### 구현 단계 (need-driven)

**Phase 1 — 수동 시뮬레이션부터.** 에이전트 없이 위 흐름을 수동으로 모방해본다. 수동으로도 충분하면 자동화 안 함. 학습법에도 need-driven 적용.

→ Phase 2~4는 검증 후 점진적 도입. 상세는 log-04.

### learning-lab과의 관계

learning-lab 레포의 v1~v3가 **선언(declaration)**이었다면, v4의 에이전트는 **실행(execution)**이다. 학습법이 비로소 시스템이 되는 시점.

```
learning-lab/         ← 학습법 정의
   │
   ↓ 운영 엔진
[학습 메타 에이전트]   ← v4부터 도입
   │
   ↓ 결과
[옵시디언]            ← 외화 저장소
   │
   ↓ (시간 경과)
[iq-blogger]          ← 장기 콘텐츠 양산
```

## 결론

v3까지는 **학습 본문 안의 흐름**(추측 → 답 → 인출)을 다듬는 진화였다. 본 미션 회고에서 v3의 8a/11번이 적용 비중 낮은 이유를 추적한 결과 — **본문 진입/퇴장의 구조 부재**가 원인이었다.

v4는 한 단계 위 차원에서 작동한다. 학습 본문 외부 구조를 시스템화하고, 그 시스템을 에이전트로 운영한다. A/B/C/D는 그 에이전트의 4개 부품이지, 별개 원칙이 아니다.

learning-lab의 비전 — *"학습법 자체를 학습하는 공간. 학습법은 코드처럼 버전 관리하고, 나선형으로 고도화한다"* — 이 v4에서 실행 가능한 형태가 된다. **학습법이 코드라면, 학습법 자동화도 코드다.**

다음 자기주도 학습 사이클에서 Phase 1(수동 시뮬레이션)부터. 자동화가 정말 필요한지부터 검증한다.

> **학습법은 시스템이고, 시스템은 자동화된다. 그러나 자동화는 검증 후에 한다.**
