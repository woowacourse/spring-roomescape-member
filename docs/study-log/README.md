# 학습법 버전 3

## 중간 기록

---
- 기록하고 싶은 개념이 생기면 아카이빙 문서를 작성한다
  - 키워드는 적당히 걸러서 기록한다(너무 사소한 것은 기록하지 않는다)
  - 하루에 하나 제일 중요한 걸 정해서 그것만 정리한다
  - 오늘 해결하지 못한 키워드는 삭제한다 (숙제로 만들지 말 것)
- 트러블슈팅이 발생하면 문제상황을 간단히 기록한다
  - 문제 - 시도 - 해결의 템플릿을 만들어 기록을 용이하게 한다
  - 문제상황은 디테일하게 정리하기보다는 단순 복사 붙여넣기를 통해 단순화한다
  - 기록에 5분 이상 쓰지 않는다
- 모르는 개념이 생기면 구글링을 하고, 날짜-제목-링크를 간단히 기록한다. 관련 아카이빙이 존재하는 경우 이 페이지와 연결해 기록한다
- 기록보다는 생각에 중점을 두려 노력한다

## 정리

---

- 사이클이 끝날 때마다 lms 학습 정리, 리뷰 코멘트 정리, 회고 정리를 기록한다
  - 리뷰어 코멘트 정리 시 AI를 사용하지 않는다
- 매일 회고(오늘 할(한) 일, 내일 할 일, 궁금한/필요한/알아낸 것, 느낀점 기록)를 진행했다
  - 할 일을 적을 때는 한 목록을 ‘20분 안에 성공 여부를 알 수 있는 가장 작은 조각’으로 쪼갠다.

### 할 일 분할 단위

> 이 일을 끝내고 났을 때, 내가 '작동하는 무언가'를 보여주거나 '하나의 명확한 원칙'을 말할 수 있는가?
>
- **최소 단위의 성공(Smallest Testable Success)**
  - **단위:** **[기능 조각 + 테스트 통과]**
  - **어떻게 나누나**
    - (나쁜 예) "로그인 기능 구현하기"
    - (좋은 예) "잘못된 비밀번호 입력 시 401 에러 반환하는 테스트 코드 작성 및 통과"
  - **왜 효과적인가? (전문가 멘탈 모델)**
    - 이것은 **'마이크로 피드백'** 전략. 뇌는 4시간 뒤의 큰 성공보다 20분마다 찾아오는 작은 성공에 훨씬 더 강력하게 반응. "성공했다"는 신호가 자주 올수록 학습 피로도는 급격히 낮아짐.
- **'인지적 델타(Cognitive Delta)' 단위**
  - **단위:** **[A-ha! 모먼트 추출]**
  - **어떻게 나누나:**
    - (나쁜 예) "JPA 영속성 컨텍스트 1장 읽기"
    - (좋은 예) "엔티티가 준영속 상태가 되었을 때 발생하는 예외 상황을 코드로 재현하고 이해하기"
  - **왜 효과적인가? (전문가 멘탈 모델)**
    - 지식의 양이 아니라 '신경망의 변화(Delta)'를 측정. 단순히 정보를 읽는 행위는 뇌에 아무런 흔적을 남기지 않지만, "아! 이게 이래서 이랬구나!"라는 지점을 정복하는 것은 뇌의 지도를 다시 그리는 작업.
- **‘버티컬 슬라이스(Vertical Slice)’ 단위**
  - **단위:** **[데이터 한 바퀴 흐름 완결]**
  - **어떻게 나누나:**
    - (나쁜 예) "오늘 Repository 계층 다 만들기"
    - (좋은 예) "단순 조회 API 하나를 DB부터 Controller까지 연결해서 포스트맨 응답 확인하기"
  - **왜 효과적인가? (전문가 멘탈 모델)**
    - 전문가는 시스템을 '전체적인 맥락(Context)'으로 이해. 수평적으로(DB만, 로직만) 나누면 전체 구조를 놓치기 쉽지만, 얇게라도 수직적으로(한 바퀴 흐름) 구현하면 시스템의 아키텍처가 뇌에 입체적으로 각인.

## 새로운 학습법

---

- **의도적 제약 (Constraint):** 기능을 구현할 때 스스로 제약을 건다
  - 예: "오늘 모든 로직에서 `else` 예약어를 쓰지 않고 다형성으로 해결해보기"
- **실패 지도 (Failure Mode):** 새로운 기술(Spring Security, Redis 등)을 배울 때 "어떻게 작동하는가"보다 "어떤 상황에서 가장 처참하게 망가지는가?"를 먼저 파악하고 기록한다
- **인지적 브릿지 (Metaphor):** 추상적인 개념을 현실의 물리적 사물에 비유하여 아카이빙 상단에 적는다
  - (예: "영속성 컨텍스트는 DB에 가기 전 들르는 임시 보관함이다")
