# Part 2: v4 비전 — 학습 메타 에이전트

## A/B/C/D를 다시 본다

v3까지는 4가지 씨앗을 별개 도구로 봤다:

| 씨앗 | 개별 정의 |
|------|----------|
| A | 키워드 1차 빠른 학습 → 계획 → 본 학습 |
| B | 옵시디언 정리 프롬프트 사전 정교화 |
| C | 옵시디언 방향그래프 + 위상정렬 |
| D | 피드백 전문가별 프롬프트 분리 |

그러나 4가지 모두 다음 공통점을 갖는다:

1. **학습 본문 외부**의 작업이다 (시작 전 또는 종료 후)
2. **반복적**이다 (매 학습 사이클마다 같은 작업)
3. **자동화 가능**하다 (옵시디언 데이터 + 프롬프트 템플릿 + 메타 정보)
4. **연결돼 있다** (A의 키워드 훑기 → C의 그래프 → B의 옵시디언 정리 → D의 전문가 피드백 — 하나의 흐름)

→ 별개 도구로 두면 매 학습마다 4번씩 의식적으로 트리거해야 한다. **하나로 묶으면 한 번의 활성화로 전체 흐름이 돈다.**

## 학습 메타 에이전트 명세

### 정체

학습 시작 전·후에만 활성화되는 보조 에이전트. 학습 본문(클로드와의 학습)에는 개입하지 않는다. **학습법의 운영 엔진**이다.

### 활성화 흐름

```
[학습 시작]
   │
   ↓ 에이전트 마크 클릭
   ↓
[Pre-learning 단계]
   ├─ 카테고리 선택: 이어할 학습 / 새 학습
   ├─ 이어할 경우 → 옵시디언에서 위상정렬된 의존 자료 + 이전 핵심 요약 가져옴 (C)
   ├─ 새 학습 → P(문제 정의) 좁히기 가이드 + 30초 추측 트리거 (8a)
   ├─ 키워드 1차 훑기 가이드 (A)
   └─ 학습 본문 컨텍스트 패키지 생성:
       - 사전 자료 (위상정렬됨)
       - 이전 요약
       - 추측 결과
       - 학습 목적 P
   │
   ↓ 컨텍스트 패키지 + 옵시디언 정리 프롬프트(B)를 클로드에 전달
   ↓
[학습 본문]
   ├─ 클로드와 학습 (에이전트는 백그라운드)
   └─ v1의 1~7, v2의 9, v3의 8a 등 적용
   │
   ↓ 학습 종료 후 에이전트 재활성화
   ↓
[Post-learning 단계]
   ├─ 학습 대화 수집
   ├─ 옵시디언 정리 (B의 프롬프트 자동 실행)
   │   ├─ 폴더 결정 (카테고리 매핑)
   │   ├─ 결론 추출 (10번 원칙)
   │   └─ 그래프 업데이트 (방향 그래프, 의존 관계 추가)
   ├─ 다음 학습 후보 제시 (위상정렬상 다음 노드)
   ├─ (선택) 24시간 후 인출 알림 설정 (11번)
   └─ (선택) 학습 전문가 피드백 트리거 — 사용자가 선택 (D)
```

### 해결되는 v2/v3의 충돌 지점

| 충돌 지점 (v2/v3에 명시) | 에이전트의 해결 |
|----------------------|---------------|
| 외화의 지속 비용 (정리 시간 아까움) | 자동 정리 → 비용을 거의 0으로 낮춤 |
| 학습 키워드 위상 정렬 (사전 지식 의존성) | 그래프가 자동 관리 |
| AI 배제(3번) vs AI 활용(8b) 결정 부담 | 에이전트가 단계별로 모드 분리. 본문에선 클로드, 시작/종료에선 에이전트 |
| 8a 추측의 진정성 | 에이전트가 추측을 강제 단계로 둠 |
| 11번 24h 인출의 지속 가능성 | 알림 자동화로 의식적 결정 비용 제거 |
| 자료 부족 영역의 시작점 | A의 키워드 1차 훑기로 약간 완화 (완전 해결은 아님) |

### 시스템 관계도

```
learning-lab/      ← 학습법 정의 (v1, v2, v3, v4)
   │
   ↓ 운영 엔진
   │
[학습 메타 에이전트]   ← v4부터 도입. learning-lab의 실행체.
   │
   ├─ 입력: 옵시디언 데이터 + 학습 의도
   ├─ 출력: 컨텍스트 패키지 / 자동 정리된 메모 / 그래프 업데이트
   │
   ↓
[클로드와의 학습 본문]
   │
   ↓ 결과
   │
[옵시디언]            ← 외화 저장소
   │
   └─ (시간 경과 후) iq-blogger가 콘텐츠로 양산
```

learning-lab의 v1~v3가 **선언(declaration)**이었다면, v4의 에이전트는 **실행(execution)**이다. 이게 학습법이 진짜로 시스템이 되는 시점이다.

### 구현 단계 (점진적 도입)

한 번에 다 만들지 않는다. need-driven으로 단계화:

**Phase 1 — 수동 시뮬레이션** (최우선)
- 에이전트 없이 사용자가 위 흐름을 수동으로 따라본다
- 옵시디언 정리 프롬프트(B), 카테고리 선택(C), 30초 추측(8a)을 종이/노션에서 모방
- **목적**: 에이전트가 정말 필요한지 검증. 수동으로도 충분하면 굳이 자동화 안 함.

**Phase 2 — 프롬프트 모음 + 옵시디언 템플릿**
- B의 정리 프롬프트, D의 전문가별 프롬프트를 파일로 정리
- 옵시디언에 카테고리별 인덱스 노트 만듦
- **목적**: 에이전트의 입력이 될 데이터 구조 확정.

**Phase 3 — 미니 에이전트** (Cursor / Claude Code 기반)
- Pre-learning 자동화: 카테고리 선택 → 사전 자료 패키지 출력
- Post-learning 자동화: 대화 정리 → 옵시디언 폴더에 저장
- **목적**: 핵심 흐름이 작동하는지 검증.

**Phase 4 — 전체 시스템**
- 그래프 업데이트, 위상정렬, 11번 알림까지 통합
- iq-blogger와 연결 (장기 결론 외화)

각 단계에서 검증하고 다음 단계로. 한 번에 큰 시스템 짓지 않는다.

### 미해결 충돌 지점 (v4 도입 후 검증할 것)

- **모드 분리 가정의 검증**: "미션 사이클" vs "자기주도 학습"의 모드를 에이전트가 잘 구분하는가? 둘의 경계가 흐릴 때(예: 깊이 학습 라운드)는?
- **수동 vs 자동의 임계점**: Phase 1에서 수동 시뮬레이션이 충분히 작동하면 Phase 2~4가 불필요할 수도. 자동화가 가치를 만드는 시점이 언제인가?
- **에이전트가 학습법을 화석화시킬 위험**: 도구가 굳어지면 학습법 진화가 멈춘다. v5가 등장할 여지를 어떻게 남길 것인가?
- **추측의 형식화 위험**: 8a가 강제 단계가 되면 "잘 모르겠음"만 적고 넘어가는 형식적 응답이 늘어날 수 있음.

# Part 3: 결론

이번 미션에서 받은 가장 큰 학습은 코드가 아니라 **학습법 자체였다**.

v3까지의 학습법 진화는 *"학습 본문 안에서 무엇을 어떻게 하는가"*에 집중했다. v4의 본질은 한 단계 위 — *"학습 본문 외부의 구조를 어떻게 시스템화할 것인가"*다. A/B/C/D는 그 시스템의 4개 부품이고, 학습 메타 에이전트는 그 부품을 묶은 운영 엔진이다.

learning-lab 레포의 README에 명시된 비전("학습법 자체를 학습하는 공간. 학습법은 코드처럼 버전 관리하고, 나선형으로 고도화한다")이 v4에서 비로소 **실행 가능한 형태**가 된다. 학습법이 코드라면, 학습법 자동화도 코드다.

다음 자기주도 학습 사이클에서 **Phase 1(수동 시뮬레이션)**부터 시작한다. 자동화가 정말 필요한지부터 검증한다. 학습법에도 need-driven 원칙을 적용한다.

> **학습법은 시스템이고, 시스템은 자동화된다. 그러나 자동화는 검증 후에 한다.**
