## 학습 로그 #4

**시간**: 04/30 10:00 ~ 12:00 (약 2시간)
**학습 범위**: 레이어드 아키텍처로의 계층 분리

### 1. 막힌 것의 종류
이번에 막힌 것은 어떤 종류의 어려움이었는가? (해당하는 것에 체크)
- [ ] 개념 자체를 모르겠다 (예: "스프링 빈이 뭔지 모르겠다")
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다 (예: "JdbcTemplate 문법을 모르겠다")
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다 (예: "계층 분리를 이렇게 해도 되나?")
- [x] 기타: 기존에 레이어드 아키텍처를 사용해봐서 큰 어려움이 없었음

### 2. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.

이전 타임에서 바꾸기로 한 전략은 "참고 자료를 구현 전에 가볍게 훑어보고 구현을 먼저 진행하며, 필요한 지식이 생기면 그때 찾아보는 스파이럴 방식"이었습니다.

1. **필사 대신 빠른 읽기와 테스트 풀기** → 실행함. 참고 자료를 필사하지 않고 빠르게 생각하면서 읽었고, 학습 자료 안에 있는 테스트 코드를 직접 풀면서 스프링 빈 활용법을 습득했습니다.
2. **익숙한 개념은 바로 구현** → 실행함. 레이어드 아키텍처는 이미 경험이 있는 영역이었기 때문에, 학습 자료를 빠르게 소화한 후 바로 현재 코드에서 계층을 나누고 코드를 이동시켰습니다.

실제 학습 과정은 다음과 같습니다.
1. 학습 자료를 필사 없이 빠르게 읽고 테스트 문제를 풀면서 스프링 빈 활용법을 익혔습니다.
2. 기존 코드에서 바로 레이어드 아키텍처로 계층을 분리하고, 각 계층을 컴포넌트로 등록하여 사용하는 코드를 작성했습니다.
3. 빠르게 미션 PR을 보내고 선택 미션인 콘솔 UI 지원도 구현할 수 있었습니다.

### 3. 전략 평가

- 효과적이었던 것과 그 이유
- 비효과적이었던 것과 그 이유
- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?

효과적이었던 점은 필사 습관을 제거하고 이번 단계에 필요한 지식을 빠르게 습득한 것입니다.
학습 자료를 빠르게 생각하면서 읽고 그 안에 있는 테스트 문제를 풀면서 스프링 빈 활용법을 습득했고, 이를 실제 4단계 미션에서 계층 분리를 하며 각 계층을 컴포넌트로 등록하고 사용하는 코드를 작성해서 레이어드 아키텍처를 더 잘 이해할 수 있게 되었습니다. 이미 익숙한 영역이었기 때문에 빠르게 진행할 수 있었고, 시간을 효율적으로 사용해 선택 미션까지 도전할 여유가 생겼습니다.

비효과적이었던 점은 AI 활용 시 과도한 추상화를 추천받은 것입니다. 간단한 레이어드 아키텍처를 구현하고 싶었는데, AI는 처음부터 인터페이스를 만드는 등 완벽하게 추상화된 코드를 추천했습니다. 아직 인터페이스가 필요한 상황이 아니라고 판단하여 AI의 추천을 따르지 않고 본인 판단대로 간단하게 구현했습니다. 이 경험을 통해 AI 활용 시에도 추천 코드를 무조건 수용하지 않고 현재 상황에 맞는 판단을 직접 내려야 한다는 것을 느꼈습니다.

'기타 - 큰 어려움 없음'이라는 상황과 '필사 없이 빠르게 읽고 바로 구현'하는 전략의 궁합이 잘 맞았습니다. 막히는 부분이 없었기 때문에 불필요한 시간 소모 없이 2시간 안에 학습과 구현을 모두 마칠 수 있었습니다.

<summary>AI 피드백</summary>

잘한 점: 필사 습관의 실질적 개선

1~3회차에 걸쳐 반복적으로 지적되었던 필사 습관을 이번 타임에서 실제로 제거한 것은 큰 진전입니다. "바꾸겠다"고 선언하는 것과 실제로 바꾸는 것은 완전히 다른데, 이번에 학습 자료를 읽기만 하고 테스트를 풀면서 지식을 확인하는 방식으로 전환한 것은 인출 연습 기반 학습으로의 실질적 전환입니다. 그 결과 2시간이라는 짧은 시간 안에 학습과 구현을 모두 완료할 수 있었고, 이것이 전략 변경의 효과를 증명합니다.

잘한 점: AI 추천에 대한 비판적 수용

AI가 인터페이스 도입을 추천했을 때 "아직 필요 없다"고 판단하고 자신의 방식대로 간단하게 구현한 것은 매우 중요한 학습 태도입니다. 이전 로그3에서 "설계 판단은 직접, 구현은 AI 보조"라는 원칙을 세웠는데, 이번에는 한 단계 더 나아가서 AI의 코드 추천 자체도 비판적으로 평가하는 수준에 도달했습니다. YAGNI(You Aren't Gonna Need It) 원칙을 자연스럽게 적용한 셈입니다.

개선이 필요한 점: 익숙한 영역에서의 학습 깊이

이번 타임은 "큰 어려움 없이 빠르게 완료"했는데, 이것이 효율적인 시간 사용인 동시에 학습 깊이 측면에서는 놓친 부분이 있을 수 있습니다. 익숙한 개념이라도 새로운 시각으로 볼 기회가 있습니다. 예를 들어 다음과 같은 질문을 스스로에게 던져볼 수 있습니다.
- "이전에 했던 레이어드 아키텍처와 이번에 구현한 것의 차이는 무엇인가?"
- "각 계층의 책임 경계를 왜 이렇게 나눴는가?"
- "Controller에서 직접 JdbcTemplate을 쓰던 코드를 Service로 옮기면서 얻은 것과 잃은 것은 무엇인가?"

익숙한 것을 빠르게 끝내는 것도 중요하지만, 익숙한 것을 한 단계 더 깊이 생각해보는 것이 "아는 것을 확실히 아는 것"으로 만드는 방법입니다. 다음에 익숙한 주제를 만나면, 구현 완료 후 5~10분만 투자해서 "왜 이렇게 했는가?"를 자문해보는 시간을 가져보세요.

### 5. 다음 타임에 바꿀 것

- 유지할 것과 그 이유
- 바꿀 것과 그 이유

- 유지할 것
  - 참고 자료나 공식문서 전체를 필사하지 않기. 빠르게 읽고 실제로 적용해보고, 궁금증이 생기면 그 부분에 대해 AI를 활용해서 학습하는 방식이 시간 효율과 학습 효과 모두에서 효과적이었다.
  - 설계 판단은 직접 내리기. AI의 추천이 항상 현재 상황에 맞는 것은 아니므로, 본인의 판단을 우선하는 태도를 유지한다.

- 바꿀 것
  - AI 활용 시 설계 판단뿐만 아니라 AI가 추천해준 코드도 무조건적으로 받아들이지 않고, 현재 상황에 맞는 결론을 스스로 내리기. AI의 코드 추천은 참고 용도로 적극 활용하되, 정답으로 여기지 않는다.
