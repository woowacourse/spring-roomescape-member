## 3. 전략 평가

"학습 테스트 레포지터리를 만들고 수행하기"는 이번 단계에서 고려하기 어려웠기에 범용적으로 사용하기 무리가 있다고 보고, 대신 학습 방법 선택지 정도로 고려하면 좋을 거 같다.
마인드 맵은 쌓여진 기술 부채들을 탐구할 떄는 유효할 수 있겠으나, 시간 안에 구현이 우선인 입장에서 마인드 맵까지 그려볼 필요성과 여유는 느끼지 못 했다.

### 효과적이었던 것과 그 이유

"현재 목표와 관련이 없으면 따로 기록하고 넘어간다"는 유효했다.
지속적으로 "내가 무엇을 목표로 하는가?"를 되짚어 볼 수 있었고, 의도치않게 깊어지는 것을 경계할 수 있었다.

### 비효과적이었던 것과 그 이유

이번 단계에서는 제시된 학습 테스트 레포지터리가 없었다. 따라서 사용해 볼 기회가 없었기에 범용적 사용이 어렵다고 느꼈다.

### 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?

현재 목표는 "일단 돌아가는 코드"였다.
나선형 학습 첫 사이클이라 계층 분리 등의 나만의 기준을 세우기는 어렵다고 생각이 들었기에, 막혔다고 표시는 했지만 나름 괜찮았던 거 같다.

## 4. AI 피드백

```shell
###" 큰 어려움은 없었다"가 정말일까

"구현 시 어느정도 감은 잡을 수 있어서 큰 어려움은 없었기에 적용해 볼 기회는 많지 않았다."

이거 살짝 의심해봐요. 143분 동안 구현하면서 정말로 막힘이 없었나요? 

KeyHolder / PreparedStatement 작성법
커밋 컨벤션 (chore vs feat)
패키지 구조 기준
스택트레이스 읽기 + FK 순서 문제
@Repository vs @Component
findById 반환을 Optional로 vs 예외
LocalTime 포매팅 + orElse vs orElseGet
SQL alias 충돌
```

생각해보니 이런 것들을 기록하긴 했다. 그런데 의식하지 않은 걸 보니 내가 접하는 마찰을 과소평가하고 있던 거 같다.
