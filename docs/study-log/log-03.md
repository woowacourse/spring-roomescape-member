### 2. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
    - 응용 상황에서 다시 막히는 것을 방지하기 위해서 코드 구조만 이해하는 데서 그치지 않고, 일부 코드에 대해 "왜 이렇게 동작하는지" 설명해보는 과정을 추가한다.
      스스로 이해하는 힘을 키우기 위해 AI를 사용하기 전에 짧게라도 먼저 시도해보고, 이후에 활용하는 방식으로 보완한다.
    - 시간 관계상 빠르게 코드를 작성해야했기 때문에 AI를 사용하기 전에 짧게 시도해보는 과정은 하지 못했다.

- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.
    - 코드를 작성하는 과정은 이전 미션2와 비슷하게, 작성해야 하는 코드를 AI에게 예제 코드 제공을 부탁했다. AI가 준 예제 코드로 개념을 학습했다.
    - 2개의 테이블로 데이터를 관리하면서 프로그램이 점점 커지고게 되었고, dto도 도입하게 되었다.
      파일도 여러 개로 분리해서 사용하게 되었고, 그만큼 관리하기 어려웠다. 그런데 파일별로 어떻게 관리해야 하는지, 어떤 책임을 가져야 하는지
      개념만 들으면 이해가 가지만, 실제로 적용하기는 쉽지 않았다. 특히 이 부분은 내 애플리케이션 전체를 AI에게 줄 수 없기 때문에, 다른 크루에게
      내 애플리케이션 흐름과 계층 별 역할에 대해 어떻게 하면 좋을 지 요청했다.

### 3. 전략 평가

- 효과적이었던 것과 그 이유
    - AI는 하기 어려운 내 애플리케이션 전체 폴더의 흐름과 내용을 다른 크루에게 부탁해서 어떻게 관리하면 좋은 지 도움을 청했다.

- 비효과적이었던 것과 그 이유
    - 시간 압박으로 인해 예제를 보고 거의 복사 붙여넣기 하다 싶이 코드를 구현한 적도 있었다.
      당장 구현은 가능하지만 학습 효과는 매우 떨어졌다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
    - 계층 나누기와 테이블 연결은 단순 문법이 아닌 객체의 책임에 관한 문제이다.
      AI의 설명은 어떻게 하면 좋을 지 이론적인 해결책은 주었으나,
      전체적인 계층 구조를 해결하는 데는 동료와의 소통이 훨씬 더 효과적이었다.

### 4. AI 피드백

메타인지와 실행의 간극: '설명해보기'라는 좋은 전략을 세웠음에도 "시간 관계상" 생략했다는 점은, 학습의 목적이 '기능 구현(Output)'에만 쏠려 있음을 의미합니다. V3
시스템의 핵심인 '지식 증발 방지'를 위해서는 [구현 1단위 → 5분 복기]의 강제적인 규칙이 필요해 보입니다.
사회적 학습(Social Learning)의 활용: 내 프로젝트 전체를 파악하지 못하는 AI의 한계를 인지하고, 즉시 크루(동료)를 찾아가 '흐름'을 점검받은 것은 매우 훌륭한
전략적 유연성입니다. 혼자 끙끙대며 매몰되는 시간을 효과적으로 단축했습니다.
추상화의 함정: "개념은 이해가 가지만 적용이 어렵다"는 것은 지식이 '추상적 수준'에 머물러 있다는 신호입니다. 이때는 AI에게 코드를 짜달라고 하기보다, "이 계층에서 저
계층으로 데이터를 넘길 때의 데이터 흐름도(Sequence)"를 그려달라고 하여 시각적 이해를 먼저 확보하는 것이 좋습니다.
