# 학습 로그 #3

**시간**: 04/29 20:30 ~ 22:00 (약 90분, 4단계) + 04/30 09:00 ~ 17:30 (약 510분, 깊이 학습 라운드) — 합계 약 600분
**학습 범위**: 4단계 계층 분리 + 깊이 학습 라운드 (리팩토링, JDBC 내부)

## 1. 막힌 것의 종류

- [x] **코드는 돌아가는데 이게 맞는 건지 모르겠다** — 4단계 자체가 설계 판단의 벽. 어디를 분리할지, 일급 컬렉션은 둘지, Record로 갈지, setter는 지울지 판단 흐림.
- [x] **개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다** — JDBC/JdbcTemplate 내부, KeyHolder의 out parameter 패턴, ResultSet 동작

4단계 끝낸 후, 미뤄뒀던 깊이 학습을 본격적으로 돌리는 라운드였다. 학습 비중으로는 4단계 자체보다 **깊이 학습 라운드가 더 컸다**.

## 2. 이번 타임의 학습 전략

### 4단계 — v1의 2번(AI 배제 설계) 부활

레이어드 아키텍처 설계는 본인이 OOP 미션에서 다뤄본 영역. 본인 머리로 그림을 그렸다:

```
Controller  →  Service  →  DAO  →  DB
                ↘ Domain ↗
```

이걸 그린 다음 AI에게 검증 받음 (v1의 3번 회복 — 답 금지, 검증 위주). "Service의 add가 DAO를 두 번 호출하는 게 맞아?" 같은 질문을 받고 의도 답했음.

### 깊이 학습 라운드 — 한 결론씩, 결론 명시 강하게

미션 통과 후 미뤄둔 호기심을 한 줄로 풀어냈다. 진행한 깊이 학습:

1. **SQL 상수화 일관성** — "왜 어떤 건 상수고 어떤 건 인라인이지?" 의문 → 모두 상수화
2. **RowMapper 위치 컨벤션** — 필드인데 생성자 아래 있던 것 → 필드 → 생성자 → 메서드 순으로 정리
3. **일급 컬렉션 도입 고민** — 행위 없으면 가치 없음 → 도입 안 함 (need-driven)
4. **RowMapper / ResultSet / KeyHolder / JDBC / JdbcTemplate** — 한 개념씩 한 라운드. 끝나면 결론 한 줄.
5. **by-value vs by-reference** (자바엔 by-reference 없음) — out parameter 패턴 이해의 전제
6. **Record 선별 도입** — 변경 가능성 없는 것만 (`ReservationTime`, DTO들) / 미래 상태 자랄 가능성 있는 `Reservation`은 일반 클래스 유지
7. **setter 제거 + final 필드** — 현재 미사용 + 캡슐화 보강
8. **INSERT-then-SELECT 비효율 제거 + 시간 검증** — DB 왕복 자체는 같지만 책임 명확화 + 비즈니스 검증 자라남
9. **String → LocalDate/LocalTime** — 도메인 표현력 강화. Jackson/JdbcTemplate 자동 변환 활용

각 라운드는 v2의 9번(끊는 기준)과 10번(결론 명시) 짝으로 진행했다.

### v3의 11번(24시간 인출)은 이번에도 미적용

라운드가 너무 많아서 24시간 후 빈 종이 인출까지 적용할 여유가 없었음. **이게 v3의 한계 — 미션 사이클에는 11번이 너무 부담**이라는 결론.

## 3. 전략 평가

### 효과적이었던 것

- **본인 의문에서 출발하는 라운드 구조**: 매 라운드의 출발점이 본인 의문이라 학습이 깊어짐. "왜 어떤 SQL은 상수고 어떤 건 인라인?", "Object 왜 쓰지?", "int 형 맞아?" 같은 사소한 질문이 KeyHolder, ResultSet, JDBC 자체 이해까지 끌고 옴.
- **결론 명시 (10번) 강화**: 매 라운드 끝에 한 줄 결론 + 커밋 메시지로 박음. "여기는 끝, 다음 스텝" 구조가 명확해짐.
- **i+1 끊는 기준 (9번)**: 일급 컬렉션 도입을 "행위 없으면 가치 없음"으로 끊은 게 좋은 예. 패턴 연극(pattern theater)을 피함.
- **v1의 4번(바텀업) + 9번(i+1) 짝**: RowMapper → ResultSet → JDBC → JdbcTemplate 순으로 한 칸씩 짚어 올라옴. DFS로 빠지지 않고 본 문제로 잘 돌아옴.

### 비효과적이었던 것

- **v3의 11번(24시간 인출) 미적용**: 라운드 9개를 연달아 돌렸는데, 24시간 후 빈 종이 인출 안 함. **진짜 이해했는지 검증 없이 다음으로 넘어감**.
- **8a(30초 추측) 미적용**: 깊이 학습 라운드는 자기주도 학습 영역인데도 8a를 안 함. AI에 그냥 답 받음. 추측을 적었다면 본인 모델이 어디서 틀렸는지 더 명확히 보였을 것.

### 막힌 것의 종류와 전략의 궁합

"설계 판단의 벽"에는 v1의 2번(AI 배제 설계)이 잘 작동. 익숙한 OOP 영역이라 가능했음. **만약 익숙하지 않은 영역이었다면 같은 전략이 안 통했을 것** — 영역 익숙도에 따라 전략 선택해야 함이 다시 확인됨.

## 4. AI 피드백

이번 미션 + 깊이 학습 라운드에서 AI에게 받은 핵심 인사이트:

- **"필요할 때 도입한다 (Need-Driven)"** — 일급 컬렉션, 응답 DTO, 트랜잭션 모두 행위/필요가 없으면 도입 가치 없음. 추상화는 비용이 있고, 비용을 정당화하는 건 행위.
- **타입 시스템에 검증을 맡겨라** — `String date`를 `LocalDate date`로 바꾸니 Jackson이 검증을 자동으로 해줌. 코드에서 검증하는 것보다 타입에 책임을 넘기는 게 안전.
- **out parameter 패턴은 자바스럽지 않지만 KeyHolder 컨텍스트에선 합리적 타협** — JDBC API의 `int update()` 시그니처 제약 + 람다 콜백 + 다양한 키 형태 때문에 Tuple/결과 객체 대안보다 깔끔.
- **체크 예외 SQLException은 메서드 시그니처에 전염된다** — JdbcTemplate이 이를 런타임 예외로 변환해 도메인 메서드 시그니처를 깔끔하게 만든다. Spring의 `DataAccessException` 계층이 핵심.

## 5. 다음 타임에 바꿀 것

### 유지할 것
- **본인 의문에서 출발하는 라운드 구조** — 모든 라운드의 출발점.
- **결론 명시 (10번) + i+1 끊는 기준 (9번)** 짝 — 깊이는 가되 본 문제로 돌아오는 구조.
- **v1의 2번(AI 배제 설계)을 익숙한 영역에서 부활** — 4단계처럼 익숙한 영역에선 본인 머리로 먼저.

### 바꿀 것 (= v4 씨앗)

이번 미션 회고에서 발견한 v4 시도 4가지:

#### A. 키워드 1차 빠른 학습 → 계획 → 본 학습
처음부터 깊이 들어가지 말고, **아는 선에서 키워드를 1차로 훑어 지도를 그린 뒤** 그걸 기반으로 학습 계획을 짜고 본 학습으로 진입. 시간 측면에서 효율적이며, P(문제 정의)도 더 뾰족해진다.

8a(30초 추측)와 결합 가능: 추측 → 키워드 1차 훑기 → 계획 → 본 학습.

#### B. 옵시디언 정리 프롬프트 사전 정교화
AI에 학습 들어가기 전, **결과를 어떻게 옵시디언에 정리할지 프롬프트를 미리 정교하게 만든다**. 새 채팅에 그 프롬프트를 보내고 학습 진행 후, 끝낼 때 그 프롬프트로 정리 요청.

10번(결론 명시)의 외화 비용을 낮추는 도구.

#### C. 옵시디언 방향그래프 + 위상정렬
학습 키워드 의존성을 옵시디언 그래프로 시각화하되 **방향그래프**로 둔다. 다음 학습 시작 시 위상정렬된 사전 자료 + 핵심만 담은 이전 학습 요약을 받고 시작 → 어디서부터 이어할지가 명확해짐.

v2의 "남은 충돌 지점" 중 **학습 키워드 위상 정렬**의 직접적 해결안.

#### D. 피드백 전문가별 프롬프트 분리
v3의 AI 학습 전문가 피드백 메커니즘을, 전문가 1명당 프롬프트를 미리 분리해 저장. 학습 상황에 따라 어떤 전문가에게 받고 싶은지 그때그때 유연하게 선택.
