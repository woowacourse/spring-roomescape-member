### 2. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?

이전 단계에서 정리한  
**“키워드 기반 개념 학습 → 코드 적용 → AI 검증” 전략을 유지**했다.

특히 이번에는  
단순 구현보다는 **구조(Controller-Service-DAO 역할 분리)**와  
**데이터 흐름**이 적절한지 확인하는 데 집중했다.

---

- 실제로 어떻게 학습했는지 디테일한 과정

1. 시간 관리 기능 요구사항을 기준으로 필요한 구조 설계 시도
2. Controller → Service → DAO 흐름을 직접 구성해봄
3. 구현 과정에서 발생한 에러 로그를 기반으로 원인 분석 시도
4. 원인이 명확하지 않은 경우 AI에게 로그 해석 및 방향성 질문
5. 특히 DB 관련 에러 (ID null, insert 구조 등)를 수정하면서  
   데이터 흐름과 DB 설계에 대한 이해 보완
6. 구현 후 전체 흐름이 맞는지 AI에게 검증 요청

---

### 3. 전략 평가

#### 효과적이었던 점

- 단순히 기능 구현이 아니라  
  **전체 흐름(Controller → Service → DAO → DB)을 이해하는 데 도움이 되었다.**
- 에러를 단순히 해결하는 것이 아니라  
  **왜 발생했는지 추적하려고 한 점이 의미 있었다.**
- 이전보다 AI에게  
  **정답이 아닌 원인과 방향을 묻는 방식으로 질문하게 되었다.**

---

#### 비효과적이었던 점

- 에러가 발생했을 때  
  **로그를 충분히 분석하기 전에 AI에 의존하는 경우가 있었다.**
- 특히 `ApplicationContext failure`와 같은 메시지에서  
  **진짜 원인을 놓치고 결과 메시지에 집중하는 실수가 있었다.**
- 구조에 대한 확신이 부족해서  
  **이미 작성한 코드를 여러 번 수정하며 시간 소모가 발생했다.**

---

#### 전략과 문제 유형의 궁합

- 이번 문제는 “구조가 맞는지에 대한 확신 부족”이었기 때문에  
  단순 개념 학습만으로는 해결되지 않았다.
- 대신  
  **구현 → 검증 → 수정 반복 과정이 더 중요했다.**
- 따라서 기존 전략은 기본 틀로는 유효했지만,  
  **디버깅과 검증 과정이 더 강화될 필요가 있었다.**

---

### 4. AI 피드백

- 현재 학습 흐름(개념 → 적용 → 검증)은 좋은 방향이다.
- 다만 다음이 부족하다는 피드백을 받았다.

  → **에러 로그를 직접 해석하는 과정**

- 특히:
    - 결과 메시지에 집중하지 말고
    - `Caused by`, `SQL Exception` 등  
      **실제 원인을 먼저 찾는 습관이 중요하다**

---

- AI를 활용한 구조 및 개념 검증

→ 단순 구현이 아니라 **이해 중심 학습에 효과적이었기 때문**

---

→ 현재는 디버깅 과정이 부족하여  
문제 해결 능력이 제한될 수 있기 때문
