```
# 실제 학습 과정
- 예약 시간이 별도 테이블(reservation_time)로 분리되면서 reservation이 time_id를 참조하도록 테이블 구조를 변경했다.
- reservation을 생성할 때 time을 직접 넣는 것이 아니라, 이미 존재하는 reservation_time의 id를 참조해야 한다는 것을 이해했다.
- 고정된 테스트에서는 reservation 생성 시 timeId=1을 사용하고 있었다.
  따라서 테스트를 통과하려면 예약 생성 전에 id=1인 reservation_time 데이터가 존재해야 했다.
- 동시에 시간 관리 API 테스트에서는 POST /times 이후 GET /times의 size가 1이어야 했다.
  이 때문에 data.sql로 기본 시간 슬롯을 여러 개 넣으면 테스트가 깨질 수 있었다.
- 그래서 시간을 어떤 방식으로 저장해야 하는지 고민했다.
  1. data.sql로 기본 시간을 미리 넣을지
  2. 테스트 안에서 POST /times를 먼저 호출할지
  3. save 시 같은 시간이 있으면 기존 값을 반환하게 할지
  4. 중복이면 예외를 던지게 할지
  5. 중복 insert를 SQL에서 조용히 무시하게 할지
- 문제는 저장 정책에 대한 고민이 너무 깊어졌다는 점이다.
  테스트를 통과시키기 위한 정책인지, 실제 서비스 요구사항에 맞는 정책인지가 섞였다.
- 예를 들어 같은 시간으로 POST /times 요청이 다시 들어왔을 때
  "중복 오류를 내는 것이 맞는지", "기존 값을 반환하는 것이 맞는지", "조용히 무시하는 것이 맞는지"를 오래 고민했다.
- 하지만 이번 단계의 핵심은 정책을 완성하는 것이 아니라, 고정된 테스트를 만족하는 최소 조건을 찾는 것이었다.
- 결국 이번 시간에는 문제를 완전히 해결하지 못했고,
  테스트가 요구하는 상태와 실제 저장 정책을 분리해서 봐야 한다는 점을 확인한 상태로 마무리했다.

# 후기
- 이번에 어려웠던 부분은 외래키 자체나 삭제 정책이 아니었다.
  연관관계가 있을 때 데이터가 먼저 존재해야 한다는 점은 이해하고 있었다.
- 실제로 어려웠던 것은 "테스트를 만족시키기 위해 save가 어떤 정책을 가져야 하는가"였다.
- 같은 시간 저장 요청을 중복으로 볼지, 기존 데이터 조회로 볼지, 실패로 볼지에 따라 코드 구조가 달라졌다.
- 그런데 이 고민을 실제 요구사항 수준까지 확장하다 보니 너무 깊어졌다.
- 이번 로그에서는 해결보다는 문제를 잘못 깊게 판 지점을 확인한 것이 더 컸다.
```

```
# 효과적이었던 것
- 테스트 실패를 단순 에러로 보지 않고, 저장 정책 문제로 바라본 것

  중복이면 실패할 수도 있고, 기존 값을 반환할 수도 있고, 무시할 수도 있다.
- 이 차이를 고민하면서 save 메서드가 단순 insert 이상의 의미를 가질 수 있다는 점을 알게 되었다.
- 테스트를 통과하려면 단순히 DB에 데이터를 넣는 것이 아니라,
  테스트가 기대하는 초기 상태와 저장 결과를 맞춰야 한다는 것도 확인했다.
```

### 4. AI 피드백

- 자신의 학습 전략에 대해 AI 학습 전문가에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록

```
# 나의 전략 분석
- 기존 전략은 "구현 문제는 빠르게 해결하고, 환경/구조 문제는 한 단계 더 이해한다"는 방식이었다.
- 이번에는 문제를 환경/구조 문제로 보고 깊게 들어갔지만,
  실제 핵심은 "테스트를 만족시키기 위한 저장 정책을 어떻게 정할 것인가"였다.
- 저장 정책에 대한 고민은 필요했지만, 테스트를 통과하기 위한 최소 정책과 실제 서비스 정책을 분리하지 못했다.
- 그래서 문제 해결보다 정책 고민에 시간이 많이 쓰였다.

# AI 학습 전문가의 피드백
- 이번 문제의 핵심은 save 정책 자체가 아니라, 여러 주제가 동시에 팝업됐을 때 분리하지 못한 점이다.
- 정책 고민은 필요하지만, 한 번에 모든 정책을 확정하려 하면 학습 범위가 커진다.
- 따라서 문제가 복잡해질 때는 먼저 다음처럼 나누는 것이 좋다.
  1. 지금 목표 달성에 필요한 문제
  2. 테스트 때문에 생긴 문제
  3. 실제 서비스 정책으로 나중에 결정할 문제
  4. DB 초기화나 기본 데이터처럼 환경 설정에 가까운 문제
- 이 중 현재 목표에 직접 필요한 문제부터 해결하고, 나머지는 기록해두는 방식이 효율적이다.

# 유용했던 제안
- 정책 고민을 두 단계로 나누는 것이 필요하다.
  1. 테스트 계약을 만족하는 최소 정책
  2. 실제 서비스에서 유지할 도메인 정책
- 이번 단계에서는 1번을 먼저 결정했어야 했다.
- 2번은 다음 단계나 요구사항이 더 명확해졌을 때 확정해도 된다.

# 다음에 시도할 구체적인 개선
- 정책이 필요한 문제를 만나면 바로 정답을 찾지 않는다.
- 먼저 아래 형식으로 정리한다.
  테스트가 기대하는 동작 / 최소로 필요한 정책 / 실제 서비스 정책으로 확정해도 되는가
  이 과정을 통해 테스트 통과용 판단과 도메인 정책 판단이 섞이는 것을 줄인다.
```
