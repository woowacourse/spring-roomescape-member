### 1. 막힌 것의 종류

이번에 막힌 것은 어떤 종류의 어려움이었는가? (해당하는 것에 체크)

- [ ] 개념 자체를 모르겠다
- [x] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다
- [x] 코드는 돌아가는데 이게 맞는 건지 모르겠다
- [ ] 기타

구체적으로는 다음과 같았다.

- `Reservation` 객체를 생성할 때 `ReservationTime` 객체도 함께 넣어야 하는데, 요청으로는 `timeId`만 받고 있어서 실제 `startAt`이 들어있는 `ReservationTime`을
  어떻게 만들어야 할지 막혔다.<br>
  -> 관계가 생긴 뒤 요청 DTO와 응답 객체의 정보량이 달라졌을 때, 이를 코드로 어떻게 연결해야 하는지에 대한 사용법 문제에 가까웠다.

- `ReservationTime`을 한 번 더 조회해서 응답 객체를 만들지, 아니면 `reservation`과 `reservation_time`을 조인해서 한 번에 조회할지 판단하는 데 시간이 걸렸다.<br>
  -> 구현 문법 자체보다, 어떤 흐름이 더 자연스럽고 책임이 분리되는지에 대한 설계 판단의 문제에 가까웠다.

- 외래키가 생기면서 시간 데이터가 먼저 있어야 예약을 저장할 수 있다는 점은 이해했지만, 테스트와 구현에서 이 선행 조건을 어떻게 다루는지가 처음에는 헷갈렸다.<br>
  -> 단순 SQL 문제가 아니라, 테이블 관계가 생겼을 때 테스트 데이터 준비와 구현 흐름을 함께 생각해야 한다는 점에서 시간이 걸렸다.

### 2. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?

  > 2단계에서 효과적이었던 테스트를 먼저 읽고, 작은 예제나 학습 테스트를 먼저 확인한 뒤 미션 코드에 적용하는 전략을 이번에도 유지하려고 했다.  
  > 또한 구현 후 직접 설명해보는 검증 단계를 추가해보려고 했고, 실제로 막힌 지점을 질문 형태로 정리하며 공부해보려고 했다.

- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.

  > 먼저 요구사항을 읽고, 3단계가 "시간 관리 기능 추가"와 "예약과 시간 연결" 두 단계로 나뉜다는 점을 확인했다.  
  > 그래서 바로 예약 연결부터 보지 않고, `/times` API를 먼저 구현해 시간 관리 기능을 따로 완성해보았다.

  > 이후 예약 요청이 `time` 문자열이 아니라 `timeId`를 받도록 바뀌면서, 기존 `Reservation` 객체를 그대로 요청으로 받기 어렵다는 점을 확인하고 `ReservationReq` 같은 요청
  DTO를 도입했다.  
  > 여기까지는 비교적 자연스러웠지만, 저장 후 응답으로 나가야 하는 `Reservation` 안에는 `ReservationTime` 객체가 들어가야 해서 그 부분에서 오래 막혔다.

  > 이 지점에서 바로 큰 코드 안에서만 고민하지 않고, `spring-learning-test`처럼 학습할 수 있는 작은 예제를 AI로부터 먼저 만들어보았다.  
  > 그 예제를 통해 "요청 DTO에는 `timeId`만 있지만, 응답 객체에는 실제 `startAt`이 들어있는 `ReservationTime`이 필요하다"는 점을 분리해서 이해했고, 저장 후 다시 조회하거나
  조인 조회로 완성된 객체를 가져오는 방식이 자연스럽다는 점을 확인했다.

  > 마지막으로 `reservation`과 `reservation_time`을 조인하는 조회 쿼리를 적용하고, 요청 DTO, 외래키, 응답 객체 구조가 서로 어떻게 연결되는지 다시 정리하면서 구현에 반영했다.

#### 3단계 목표 설정

> 관리자가 예약 시간을 직접 입력하지 않고 선택할 수 있도록 시간 관리 기능을 추가하고, 예약과 시간을 연결하는 과정에서 테이블 간 관계와 객체 간 의존 관계를 실제 코드로 이해하는 것이다.

#### 1. 기능 목표

- `reservation_time` 테이블을 추가하고 시간 추가/조회/삭제 API를 구현한다
- `POST /times`, `GET /times`, `DELETE /times/{id}`를 동작하게 만든다
- 기존 예약 기능에 시간 선택 개념을 연결한다
- `reservation` 테이블의 `time` 컬럼을 `time_id`로 변경한다
- 예약 추가 요청에서 `time` 대신 `timeId`를 받도록 변경한다
- 예약 조회 응답에서 `time`을 `{id, startAt}` 형태의 객체로 반환한다
- 주어진 테스트를 통과한다

#### 2. 기술 목표

- 테이블 간 외래키 관계를 이해하고 적용한다
- 객체 간 의존 관계가 생길 때 어떤 변경이 필요한지 익힌다
- `inner join`을 사용해 여러 테이블의 데이터를 함께 조회하는 방법을 익힌다
- 기존 쿼리와 API 스펙이 관계 추가로 인해 어떻게 바뀌는지 확인한다
- 새로운 테스트 도구를 추가하지 않고 JUnit과 기존 RestAssured 테스트 흐름 안에서 검증한다

#### 3. 학습 목표

- "시간 기능을 추가했다"보다 예약과 시간의 관계가 생기면서 어떤 코드와 쿼리가 함께 바뀌는지 설명할 수 있다
- 예: 시간 테이블 추가 -> 시간 API 구현 -> 예약 요청 형식 변경 -> 예약 조회 쿼리 join 적용 -> 응답 구조 변경
- 테이블 간 관계와 객체 간 관계가 서로 어떻게 대응되는지 설명할 수 있다
- 왜 문자열 시간 입력 방식에서 시간 선택 방식으로 바꾸는지 설명할 수 있다

#### 4. 미션 운영 목표

- 처음부터 완벽하게 설계하려 하기보다 시간 API를 먼저 동작하게 만들고, 그 다음 예약과 시간을 연결한다
- 막히면 전체 구조를 한 번에 보려 하지 말고 시간 기능과 예약 연결 기능을 나누어 구현한다
- 쿼리가 복잡해지면 먼저 SQL을 따로 읽고, 어떤 데이터가 필요한지 확인한 뒤 코드에 적용한다
- 한 타임이 끝나면 `log-03.md`에 아래 내용을 기록한다
- 어디서 막혔는지
- 어떤 방식으로 공부했는지
- 다음에는 무엇을 바꿀지
- 이전 단계에서 사용한 전략이 3단계의 관계 처리 문제에도 잘 맞았는지 함께 점검한다

### 3. 전략 평가

- 효과적이었던 것과 그 이유

  > 테스트를 먼저 읽고, 시간 관리 기능과 예약 연결 기능을 분리해서 본 방식이 효과적이었다.  
  > 3단계를 한 번에 다 보려고 했다면 더 복잡하게 느껴졌을 텐데, 먼저 `/times` API를 따로 구현해보니 이후 예약 연결에서 무엇이 새로 추가되는지가 더 잘 보였다.

  > 작은 학습 예제를 먼저 만들어보고 실제 코드에 적용한 방식도 매우 도움이 되었다.  
  > 특히 `ReservationReq`와 `Reservation`의 정보량 차이, 그리고 `ReservationTime`을 어디서 가져올지에 대한 고민은 큰 코드 안에서만 보면 막막했는데, 예제로 분리해서 보니
  훨씬 명확해졌다.

- 비효과적이었던 것과 그 이유

  > 완전히 비효과적이었다고 느낀 전략은 아직 없었다.  
  > 다만 처음에는 요청 DTO를 바탕으로 바로 응답 객체를 조립하려고 오래 고민했는데, 이 방식은 관계가 생긴 뒤 바뀐 정보 구조를 충분히 반영하지 못해서 시간이 조금 더 걸렸다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?

  > 이번 막힘은 관계가 생긴 뒤 요청, 저장, 응답 구조가 달라지는 과정에서 충분히 생길 수 있는 자연스러운 학습 비용이었다고 생각한다.  
  > 다만 테스트를 먼저 읽고, 작은 예제를 만들어 학습한 뒤 실제 코드에 적용하는 전략 덕분에 막힘이 길게 이어지지는 않았다.

### 4. AI 피드백

- 자신의 학습 전략에 대해 AI 학습 전문가에게 피드백을 요청하고,
  유용했던 제안 1가지 이상 기록

  > AI는 이번 막힘이 단순한 `inner join` 문법 문제가 아니라, "요청 DTO는 `timeId`만 가지지만 응답 객체는 `ReservationTime` 전체를 가져야 한다"는 정보 구조의 차이를 먼저
  분리해서 봐야 한다고 피드백했다.  
  > 그래서 관계가 생기는 단계에서는 요청 DTO, DB에 저장되는 값, 응답 객체를 같은 것으로 보지 말고 각각 따로 놓고 생각해보라는 제안이 유용했다.

  > 특히 관계가 생기는 단계에서는 문법보다 흐름과 책임이 더 중요해서, 예제로 먼저 구조를 확인하는 것이 큰 도움이 되었다.

  > 부가적인 궁금증을 따로 기록하고 구현을 우선 진행하는 방식도 유지하려고 한다.
