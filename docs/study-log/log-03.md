### 3. 전략 평가

- 효과적이었던 것과 그 이유
- 비효과적이었던 것과 그 이유
- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?

효과적이었던 점은 AI를 통한 JdbcTemplate 메소드 사용법 학습이었습니다.
JdbcTemplate에는 수십 가지 메소드가 있는데, 모든 메소드를 직접 찾아가며 현재 상황에 맞는 것을 선택하기에는 시간이 부족했습니다. AI에게 현재 상황을 설명하고 적합한 메소드를 추천받아 바로 적용하는 방식이 '코드로 어떻게 쓰는지 모르겠다'는 어려움에 잘 맞았습니다.

비효과적이었던 점은 MySQL Reserved Words 문서 학습 방식이었습니다.
참고 자료 링크를 한 번 쭉 읽고 모르는 것을 적으려는 전략을 시도했으나, 문서가 너무 길고 당장 필요하지 않은 부분도 많았으며 모르는 부분이 대부분이라 적을 것이 너무 많았습니다. "읽고 모르는 것만 적기" 전략이 모든 종류의 문서에 동일하게 적용되지는 않는다는 것을 알게 되었습니다.

### 4. AI 피드백

- 자신의 학습 전략에 대해 AI에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록

<details>
<summary>AI 피드백</summary>

잘한 점: 설계 판단과 구현 보조의 분리

DTO 도입 여부와 같은 설계 결정은 직접 내리고, JdbcTemplate 메소드 사용법 같은 도구 활용은 AI에게 맡긴 것은 매우 바람직한 학습 방식입니다. "왜 이렇게 하는가"는 본인이 판단하고, "어떻게 작성하는가"는 AI의 도움을 받는 이 분리가 유지되면, AI에 의존하면서도 설계 역량은 꾸준히 성장할 수 있습니다. 이 균형을 계속 유지하세요.

개선이 필요한 점: 문서 종류에 따른 전략 분기

"읽고 모르는 것만 적기" 전략이 MySQL Reserved Words처럼 **레퍼런스형 문서**(사전처럼 나열된 정보)에는 맞지 않았다는 발견은 중요합니다. 문서에는 크게 두 종류가 있습니다.
- **개념 설명형**: 튜토리얼, 가이드 → "읽고 모르는 것 적기"가 효과적
- **레퍼런스형**: API 문서, 예약어 목록 → 전체를 읽는 것이 아니라 **필요할 때 검색**하는 것이 적절

다음 타임부터는 문서를 열기 전에 "이 문서는 통독해야 하는 것인가, 필요할 때 찾아보는 것인가?"를 먼저 판단해보세요. Reserved Words 같은 레퍼런스 문서는 북마크만 해두고, 실제로 쿼리를 작성하다가 예약어 충돌이 발생했을 때 찾아보는 것이 훨씬 효율적입니다.

제안하는 문서 학습 루틴

1. 문서를 열면 먼저 **목차와 구조만 30초간 훑습니다**.
2. "이 문서는 지금 통독할 가치가 있는가?"를 판단합니다.
3. 통독할 가치가 있으면 → 기존 전략대로 읽고 모르는 것 적기.
4. 레퍼런스형이면 → 북마크하고 넘어가기. 구현 중 필요할 때 검색.

이 판단을 추가하면 "스파이럴 학습"의 첫 바퀴에서 불필요한 시간 소모를 줄일 수 있습니다.
