## 학습 로그 #3
**시간**: 04/29 22:00 ~ 24:00 (약 2시간)
**학습 범위**: 3단계 시간 관리

### 1. 단계 전환 체크
- 이전 단계에서 쓴 접근법을 그대로 쓸 것인가, 바꿔볼 것인가?
    - 실습 + 백지 작성 전략을 그대로 유지했다.
- 바꾼다면 왜? 바꾸지 않는다면 왜?
    - 바꾸지 않으려고 한다. 그 이유는 이전 단계에서 효과적이었기 때문이다.
- 이전 단계에서 발견한 "나에게 맞는 전략"이 새 단계에서도 통할 것 같은가?
    - 시간 관리 기능 구현은 API 설계 복습 차원이고, 백지 작성은 조인 쿼리 작성에 적용해볼 것이기 때문에 새 단계에서도 통할 것이라 생각한다.

### 2. 막힌 것의 종류
이번에 막힌 것은 어떤 종류의 어려움이었는가? (해당하는 것에 체크)
- [ ] 개념 자체를 모르겠다 (예: "스프링 빈이 뭔지 모르겠다")
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다 (예: "JdbcTemplate 문법을 모르겠다")
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다 (예: "계층 분리를 이렇게 해도 되나?")
- [x] 기타: 조인 개념이 생기면서, 쿼리를 스스로 이해하고 작성하는 과정이 어려웠다

### 3. 이번 타임의 학습 전략
- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
    - 물음표 지점 만들기와 다음 나선의 좌표를 만들기 전략이었고, 실행하지 못했다.
- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.
    - 예약 기능 구현 시 spring-mvc 학습 테스트를 이미 실습했기 때문에, 시간 관리 기능은 별도 학습 테스트 없이 복습 차원에서 바로 구현했다.
      예약과 시간 연결 과정에서 조인 쿼리 작성, rowMapper에서 ReservationTime 객체 생성 순으로 구현하면서 FK 제약에 대해 학습할 수 있었다.

### 4. 전략 평가
- 효과적이었던 것과 그 이유
    - 시간 관리 기능을 복습 차원으로 바로 구현한 것이 이미 실습한 내용이라 빠르게 진행할 수 있었다. 백지에 조인 쿼리를 작성해보는 것이 쿼리 이해에 도움이 됐다.
- 비효과적이었던 것과 그 이유
    - 물음표 지점과 다음 나선 좌표 만들기가 자연스럽게 떠오르지 않아 형식적으로 느껴졌다.
- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
    - 막힌 것이 "조인 쿼리를 스스로 이해하고 작성하는 것"이었는데, 백지 작성 전략이 효과적이었다.

### 5. AI 피드백
- 자신의 학습 전략에 대해 AI 학습 전문가에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록
    - 잘 되고 있는 것
        - 막힌 지점을 구체적으로 언어화하고 있어. "조인 쿼리 작성이 어렵다"는 걸 인식하고 백지 전략으로 연결한 게 좋아. 
        그리고 효과 없는 전략을 솔직하게 평가하고 다음에 다시 시도해보겠다는 태도도 좋아.
    - AI의 제안 1
      - 물음표 지점이 자연스럽게 안 떠오른다면, 구현 중 "왜 이렇게 했지?"라고 스스로 물어본 순간을 기록해봐. 
      FK 제약 오류를 만났을 때 "왜 시간 데이터가 먼저 있어야 할까?"라는 물음이 생겼잖아. 그게 바로 물음표 지점이야. 
      의도적으로 만들려 하기보다 구현 중 자연스럽게 생긴 의문을 포착하는 연습을 해봐.

### 6. 다음 타임에 바꿀 것
- 유지할 것과 그 이유
    - 복습 차원의 기능은 학습 테스트 없이 바로 구현하는 것과 새로운 개념은 백지 작성으로 이해를 검증하는 것을 유지한다. 효과적이었기 때문이다.
- 바꿀 것과 그 이유
    - 물음표 지점을 의도적으로 만들려 하지 않고, 구현 중 "왜?"라고 느낀 순간을 바로 메모하는 방식으로 바꿔본다.
