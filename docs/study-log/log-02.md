### 4. AI 피드백
- **공식 문서 '5분 스캐닝' 프로토콜 (Official Doc Scavenger Hunt)**
    > 구글링은 빠르지만, 지식의 파편만 얻게 되어 뇌가 '시스템의 전체 구조'를 학습할 기회를 박탈

    - 방법: 모르는 문법이 나오면 바로 구글링하지 말고, 딱 5분만 공식 문서(Spring Reference 등)에서 해당 키워드를 찾는다. 
    - 찾지 못하더라도 '목차'와 '주변 개념'을 훑는 것만으로 충분, 5분 뒤에는 원래대로 구글링을 해도 좋음.

- **'상호작용' 중심의 슬라이싱 (Interaction-Based Slicing)**
   > "Repository 구현"은 뇌에게 너무 막연합니다. 할 일을 '레이어'가 아닌 '데이터의 여행' 단위로 쪼개기

    - 방법: 할 일을 쪼갤 때 "무엇을 만드는가"가 아니라 "어떤 데이터 흐름을 완성하는가"로 적는다.
    - 예시: "DB의 ID 1번 데이터를 꺼내서 API 결과창에 띄우기"
    - 효과: 이렇게 하면 Repository의 특정 메서드 하나, Service의 한 줄, Controller의 리턴값까지 딱 필요한 만큼만 건드리게 됨.

- **'설계 보류' 주석 (The "Is this right?" Buffer)**
   > "이게 맞는 설계인가?"라는 의구심은 코딩 흐름을 끊는 가장 큰 방해 요소

    - 방법: 코드를 짜다가 "계층 분리가 맞나?" 같은 의문이 들면 멈추지 말고, 코드에 // QUESTION: 계층 분리 적절성?이라고 적어두고 일단 돌아가는 코드를 끝까지 완성한다.
    - 그 타임이 종료된 후, 혹은 리뷰 요청 시에 그 주석들만 모아서 고민
