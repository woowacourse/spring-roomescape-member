### 1. 단계 전환 체크
- 이전 단계에서 쓴 접근법을 그대로 쓸 것인가, 바꿔볼 것인가?
  - 실습으로 먼저 부딪혀본다는 전략을 그대로 유지하되, 백지에 해당 개념을 제대로 이해하고 있는지 확인하는 전략을 추가하려 한다.
- 바꾼다면 왜? 바꾸지 않는다면 왜?
  - 싫습을 하고 난 뒤에도 해당 개념을 코드로 작성하는 감은 있지만, 다른 사람에게 명확히 설명할만큼 이해하고 있지 않다고 판단해 이를 보완하기 위해서이다.
- 이전 단계에서 발견한 "나에게 맞는 전략"이 새 단계에서도 통할 것 같은가?
  - 그렇다. 실습 기반으로 먼저 부딪혀보며 해당 개념을 코드로 작성하는 방법을 이해할 수 있다고 생각하기 때문이다.

### 2. 막힌 것의 종류
이번에 막힌 것은 어떤 종류의 어려움이었는가? (해당하는 것에 체크)
- [ ] 개념 자체를 모르겠다 (예: "스프링 빈이 뭔지 모르겠다")
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다 (예: "JdbcTemplate 문법을 모르겠다")
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다 (예: "계층 분리를 이렇게 해도 되나?")
- [x] 기타: 코드로 어떻게 작성하는지는 알겠는데, 다른 사람에게 설명할 수 있는 만큼 개념을 깊이 이해하고 있지 않다 (예: "JdbcTemplate, RowMapper, KeyHolder를 명확히 설명할 수 없다")

### 3. 이번 타임의 학습 전략
- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
  - 2단계에서는 백지에 해당 개념을 제대로 이해하고 있는지 작성해보는 전략을 실행했다.
- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.
  - 구글링을 통해 해당 개념에 대해 먼저 정리한 후, 백지에 개념을 제대로 이해하고 있는지 한번 더 작성해보았다. 
  이 때 JdbcTemplate은 '스프링에서 JDBC를 사용할 때 발생하는 반복적인 코드를 자동화한 라이브러리', 
  RowMapper는 'JDBC를 사용할 때 SQL 조회 결과의 각 행을 자바 객체로 변환해주는 인터페이스', 
  KeyHolder는 'JdbcTemplate를 사용하여 데이터를 INSERT 할 때 데이터베이스에서 자동으로 생성된 키 값을 가져오기 위해 사용하는 인터페이스'로 작성했다.

### 4. 전략 평가
- 효과적이었던 것과 그 이유
  - 해당 개념을 잘 알고있는지 한번 더 확인해볼 수 있어 효과적이었다.
- 비효과적이었던 것과 그 이유
  - 개념적인 부분들을 여러 차례 글로 정리하려고 보니 이 과정이 조금 지루하게 느껴졌다.
- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  - 막힌 것이 "설명할 수 없다"는 개념 이해의 깊이 문제였는데, 백지에 써보는 전략이 적절했다. 
  단순히 코드를 동작시키는 것에서 나아가, 각 도구를 왜 쓰는지 이유를 파악하고 사용할 수 있게 되었다.
