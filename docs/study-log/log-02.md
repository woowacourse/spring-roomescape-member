### 2. 이번 타임의 학습 전략

- **이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?**
  1단계 학습 로그에서 결론 내렸던 **'투트랙(Two-track) 검색 전략 (빠른 구현을 위한 구글링 적극 병행)'**을 바로 실행에 옮겼다.
- **실제로 어떻게 학습했는지 디테일한 과정을 써보세요.**
  메모리 기반의 `AtomicLong`을 제거하고 DB 연동으로 넘어가면서, `INSERT` 쿼리 실행 직후 자동 생성된 식별자(`id`)를 응답(`ReservationCreated`)에 담아야 하는 요구사항에서
  막혔다. 공식 문서를 처음부터 뒤지지 않고, 구글에 "Spring Boot JdbcTemplate INSERT return ID"나 "JdbcTemplate KeyHolder 사용법" 등의 키워드로 직접
  검색했다. 기술 블로그와 StackOverflow에서 `GeneratedKeyHolder`와 `PreparedStatement`를 조합하는 구체적인 예제 코드를 찾아 내 코드 구조에 맞게 변형하여 적용했다.
  `application.properties`의 H2 설정(`spring.sql.init.mode` 등) 역시 구글링을 통해 빠르게 템플릿을 확보했다.

### 3. 전략 평가

- **효과적이었던 것과 그 이유**
  **압도적인 속도 개선**. 필요한 "사용법(How-to)"이 담긴 코드 스니펫을 바로 확인하니 화면 앞에서 헤매는 시간이 획기적으로 줄었다. 빠르게 테스트 코드를 통과(Green)시키고 요구사항을 충족하는 데
  매우 효과적이었다.
- **비효과적이었던 것과 그 이유**
  블로그마다 `JdbcTemplate`을 사용하는 스타일(람다식 사용 여부, 예외 처리 방식 등)이 달라서 어떤 것이 현재 내 스프링 부트 버전에 가장 적합한 모던한 방식인지 필터링하는 데 에너지가 약간
  소모되었다. 또한, 당장 동작하게 만드는 데 급급해 `KeyHolder`의 내부 구현 로직까지 깊게 이해하지는 못하고 넘어간 찝찝함이 남았다.
- **막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?**
  매우 좋았다. "코드로 어떻게 쓰는지"를 묻는 상황에서는 이미 누군가 작성해 둔 검증된 예제 코드를 보는 것만큼 직관적인 해결책이 없었다.

### 4. AI 피드백

- **자신의 학습 전략에 대해 AI 학습 전문가에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록**
    1. **블랙박스화 경계 (Beware of Black-boxing)**: 구글링을 통한 '복붙(Copy & Paste)' 식 문제 해결은 생산성을 높여주지만, `JdbcTemplate`이나
       `RowMapper` 같은 핵심 기술의 원리를 '블랙박스'로 남겨둘 위험이 있다. 동작하는 코드를 얻었다면, 리팩터링 단계에서 반드시 "왜 이렇게 동작하는가?"를 되물어야 한다.
    2. **의도적인 지연 학습 (Intentional Delayed Learning)**: 빠른 진도를 위해 구글링으로 구현 사이클을 돌렸다면, 미션 제출 전이나 주말을 이용해 적용했던 기술(
       `KeyHolder` 등)의 공식 문서를 다시 찾아보는 '학습 사이클'을 별도로 분리하라는 제안이 유용했다. 투트랙 전략의 완성을 위한 좋은 조언이었다.

### 5. 다음 타임에 바꿀 것

- **유지할 것과 그 이유**
  완벽주의를 버리고, 일단 동작하는 코드를 빠르게 만들어 피드백 루프를 돌리는 현재의 실용적인 접근 방식(구글링 병행)은 계속 유지한다.
- **바꿀 것과 그 이유**
  **주도적인 AI 도구 도입.** 단순히 구글링으로 파편화된 정보를 모으는 것을 넘어, 다음 3단계(시간 관리)나 4단계(계층 분리) 미션부터는 AI를 학습 및 구현의 보조 도구로 적극 활용해 볼 계획이다.
  단순히 코드를 짜달라고 하는 것이 아니라, "내가 짠 이 로직을 더 객체지향적으로 계층을 분리하려면 어떤 패턴이 좋을까?"와 같이 코드 리뷰를 요청하거나, 헷갈리는 특정 어노테이션의 개념을 나만의 비유로 설명해
  달라고 요청하는 등, **AI를 동료 삼아 한계를 넓히는 방식**으로 튜닝해 보겠다.
