### 2. 이번 타임의 학습 전략

이전에는 테스트 코드를 기반으로 Controller 작성법을 AI에게 직접 요청하는 방식이었다.
하지만 이번에는 다음과 같이 전략을 변경했다.
- LMS와 학습 테스트에 등장하는 핵심 키워드의 개념을 먼저 이해
- 이후 해당 개념을 실제 코드에 적용해보고
- 마지막으로 AI를 통해 내 코드와 개념이 적절한지 비교 검증

**실제 학습 과정**
1. 공식 문서를 통해 주요 개념(JdbcTemplate, Controller 등) 학습
2. 학습 테스트를 실행하며 Controller, DAO 코드 작성 시도
3. 코드 작성 중 모르는 개념은 AI에게 질문
4. 다시 코드에 적용
5. 최종적으로 AI에게 구조와 방향이 적절한지 피드백 요청
### 3. 전략 평가
**효과적이었던 점**
- 개념을 먼저 학습하고 시작하면서 Spring 구조에 대한 전체적인 흐름을 이해할 수 있었다.
- 단순히 답을 받는 것이 아니라 “왜 이렇게 쓰는지”에 대한 기준이 생겼다.
- AI에게도 정답을 요구하기보다 개념과 사용 방식 위주로 질문하게 된 점이 의미 있었다. 


**비효율적이었던 점**
- 공식 문서를 처음부터 모두 읽으려다 보니 시간이 많이 소모되었다.
- 개념을 이해한 것과 실제 코드로 구현하는 것 사이의 간극이 컸다.

**전략과 문제 유형의 궁합**
- 개념 중심 학습은 “왜 사용하는지” 이해하는 데에는 효과적이었다.
- 하지만 “어떻게 구현하는지”까지 자연스럽게 이어지지는 않았다.

그래서 중간에:
- AI에게 구현 방법을 물어보거나
- 학습 테스트를 통해 감을 잡는 과정이 필요했다.

**추가로 느낀 점**
- Controller를 처음 잘못 이해해서 이후 구조를 수정하는 데 시간이 많이 들었다.
- 특히 Controller → Service 분리 과정에서  “Spring에서는 이걸 어떻게 나누는가?”에 대한 고민이 컸다.

→ 이 부분은 AI를 통해 빠르게 해결했지만,
스스로 설계한 것이 아니라는 점에서 아쉬움이 남았다.

### 4. AI 피드백
- 개념 → 적용 → 검증 흐름은 좋은 방향이며,
특히 정답을 바로 받지 않고 개념 중심으로 질문한 점은 효과적이다.
- 다만, 에러가 발생했을 때 바로 AI에 의존하기보다
  - 로그 해석, 원인 추측, 수정 시도 

  의 과정을 먼저 거치는 것이 필요하다는 피드백을 받았다.

### 5. 다음 타임에 바꿀 것
**유지할 것**

→ 단순 구현이 아니라 이해 기반 학습에 도움이 되었기 때문

→ 현재는 디버깅 과정이 부족하여
문제 해결 능력이 AI 의존적으로 흐를 가능성이 있음

### 6. 단계 전환 체크

#### 이전 단계에서 쓴 접근법을 그대로 쓸 것인가, 바꿔볼 것인가?

이전 단계에서 사용한 **키워드 중심 개념 학습 전략을 유지할 계획이다.**

---

#### 바꾸지 않는다면 왜?

현재는 개념 이해가 충분하지 않은 상태이기 때문에,  
코드를 먼저 작성하는 방식보다는 **개념을 기반으로 접근하는 것이 더 안정적이라고 판단했다.**

코드를 먼저 작성할 경우:
- 구현 과정에서 확신이 부족해지고
- 구조가 흔들리거나
- 불필요한 오류를 만들 가능성이 높았다.

따라서 최소한의 개념 이해를 선행한 뒤 코드로 옮기는 방식이  
현재 단계에서는 더 적합하다고 느꼈다.

---

#### 이전 단계에서 발견한 "나에게 맞는 전략"이 새 단계에서도 통할 것 같은가?

대체로 통할 것이라고 생각한다.

이전 단계에서:
- Spring 구조에 대한 흐름이 조금씩 보이기 시작했고
- 코드 작성 방식에 대한 감도 생기고 있다.

이 상태에서:
- 관련 개념과 다양한 사용 방식을 추가로 학습하면  
  단순 구현을 넘어서 **왜 그렇게 작성하는지까지 이해하는 데 도움이 될 것 같다.**

다만 앞으로는:
- 개념 학습에만 머무르지 않고
- **실제 코드 적용과 에러 해결 과정까지 함께 가져가는 방향으로 보완이 필요하다.**
