### 1. 막힌 것의 종류

이번에 막힌 것은 어떤 종류의 어려움이었는가? (해당하는 것에 체크)

- [x] 개념 자체를 모르겠다

구체적으로는 다음과 같았다.

- `JdbcTemplate`, `update`, `KeyHolder`의 역할은 학습 자료를 통해 어느 정도 이해했지만, 이를 내 미션 코드의 어느 위치에 넣어야 하는지 바로 연결되지 않았다.<br>
  -> 개념 부족이라기보다, 배운 내용을 실제 코드에 적용하는 사용법의 문제에 가까웠다.

- 특히 insert 로직을 `Reservation` 객체에 둘지, `Controller`에서 직접 다룰지, `Dao`로 분리할지 판단하는 과정에서 많이 막혔다.<br>
  -> 구현 자체보다, 각 계층에 어떤 책임을 두는 것이 맞는지 확신이 없는 설계 판단의 문제에 가까웠다.

- 또한 스프링 빈 등록과 생성자 주입 개념을 몰랐다. 실제로 `ReservationDao`와 `JdbcTemplate`를 연결하는 과정에서 빈 등록은 했지만, 주입을 하지 못해 헤맸었다.

### 2. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?

  > 1단계에서 테스트를 먼저 읽고 최소 기능부터 빠르게 구현하는 방식이 효과적이었기 때문에, 이 큰 흐름은 유지하려고 했다.
  대신 2단계에서는 패키지 구조를 오래 고민하기보다 먼저 동작하는 DB 연동을 만드는 데 집중하고, `JdbcTemplate`, SQL, `KeyHolder` 같은 문법은 바로 미션 코드에 넣기보다 작은 예제와
  학습 테스트를 먼저 확인한 뒤 적용해보려고 했다.
  적용해보니 확실히 문제가 발생했을 때, 적용해야할 도구를 잘 활용했던 것 같다. 예를 들어 insertWithKeyHolder를 미리 예제를 통해 학습하지 않았다면, 구현하는데 큰 시간이 걸렸을 것 같다.
- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.
  > 먼저 요구사항 테스트를 읽고 어떤 API가 DB 기반으로 바뀌어야 하는지 확인했다. 그리고 spring-learnin-test로 간단한 예제를 통해 문법을 익혔다.
  > 이후 H2 설정과 schema를 적용한 뒤, GET 조회부터 JdbcTemplate query로 전환했다.
  > 그 다음 DELETE, 마지막으로 KeyHolder를 사용한 POST를 구현하면서 빈 등록과 생성자 주입 문제를 해결했다.

#### 2단계 목표 설정

> 예약 CRUD API를 H2 데이터베이스와 `JdbcTemplate` 기반으로 전환하고, 그 과정에서 datasource 설정, `schema.sql` 초기화, SQL 실행 흐름을 실제 코드로 이해하는 것이다.

#### 1. 기능 목표

- `build.gradle`에 `spring-boot-starter-jdbc`, `h2` 의존성 추가
- `application.properties`에 H2 콘솔과 datasource 설정 추가
- `resources/schema.sql`로 `reservation` 테이블 생성
- `GET /reservations`, `POST /reservations`, `DELETE /reservations/{id}`를 모두 `JdbcTemplate` 기반으로 전환
- 기존의 `List + AtomicLong` 방식 제거
- 예약 추가 시 DB가 생성한 `id`를 응답에 담기
- 주어진 테스트 통과

#### 2. 기술 목표

- Spring Boot에서 H2 데이터베이스를 연동하는 방법 익히기
- `JdbcTemplate`으로 `SELECT`, `INSERT`, `DELETE` 쿼리 실행하기
- `schema.sql`과 `application.properties`가 애플리케이션 실행 시 어떤 역할을 하는지 이해하기
- DB가 생성한 키를 가져오는 방법 익히기
- `KeyHolder` 또는 `SimpleJdbcInsert`의 역할과 사용 흐름 이해하기
- 새로운 테스트 도구를 추가하지 않고 JUnit 중심으로 검증하기

#### 3. 학습 목표

- "DB를 연결했다"보다 요청 1개가 들어와서 DB를 거쳐 응답이 나갈 때까지의 흐름을 설명할 수 있다
- 예: 클라이언트 요청 -> 컨트롤러 실행 -> `JdbcTemplate`으로 SQL 실행 -> DB 반영 -> 결과 응답
- 메모리 저장과 DB 저장의 차이를 설명할 수 있다
- 왜 1단계의 `List + AtomicLong`을 2단계에서 DB 방식으로 바꾸는지 설명할 수 있다

#### 4. 미션 운영 목표

- 1단계처럼 완벽한 구조보다 먼저 동작하는 DB 연동을 만들기
- 막히면 구조를 오래 고민하기보다 작은 SQL 예제나 학습 테스트로 먼저 확인하기
- 테스트를 먼저 읽고, 필요한 DB 상태와 응답 형식을 먼저 파악한 뒤 구현하기
- 한 타임이 끝나면 `log-02.md`에 아래 내용을 기록하기
- 어디서 막혔는지
- 어떤 방식으로 공부했는지
- 다음에는 무엇을 바꿀지
- 단계가 바뀌었으므로, 1단계에서 쓰던 학습 전략을 2단계에서도 유지할지 바꿀지도 함께 점검하기

### 3. 전략 평가

- 효과적이었던 것과 그 이유
  > 테스트를 먼저 읽고 필요한 동작과 응답 형식을 먼저 파악한 뒤 구현한 방식이 효과적이었다.  
  > 2단계에서도 요구사항 테스트를 먼저 확인하니, 단순히 JdbcTemplate 문법을 외우는 것보다 어떤 기능을 DB 기반으로 전환해야 하는지가 더 선명하게 보였다.

  > 또한 `JdbcTemplate`, `KeyHolder`, SQL 예제를 작은 단위로 먼저 확인하고 미션 코드에 적용한 방식도 도움이 되었다.  
  > 문법을 바로 큰 코드에 넣을 때보다, 먼저 짧은 예제로 확인한 뒤 적용하니 막힐 때 원인을 좁혀서 볼 수 있었다.

  > 부가적인 궁금증을 따로 기록하고 구현을 먼저 끝내려 한 방식도 여전히 효과적이었다.  
  > 2단계에서는 궁금한 개념이 더 많아졌지만, 그때마다 흐름을 끊지 않고 구현을 이어갈 수 있었다.

  > 빈 등록과 주입을 구분하지 못해 Dao를 완성하고도 테스트코드가 다 깨지는 이슈가 있었는데,
  > 이를 5분 내외로 짧게 고민해보고 AI를 활용한 게 효과적이었다. 아예 모르는 개념이었기에.

- 비효과적이었던 것과 그 이유
  > 완전히 비효과적이라고 느낀 전략은 아직 없었다.
  > 다만, 공부한 걸 제대로 확인할 수 있는 루틴은 있엇으면 좋겠다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  > 이번 막힘은 특정 전략이 잘못되어서 생긴 문제라기보다, 2단계처럼 새로운 개념과 구조가 함께 들어오는 과정에서 충분히 발생할 수 있는 정상적인 학습 비용에 가까웠다고 생각한다.
  > 다만 테스트를 먼저 읽고, 작은 예제와 AI 설명을 통해 필요한 개념을 바로 적용해보는 전략 덕분에 막힘이 길게 이어지지는 않았고, 다시 구현 흐름으로 돌아오는 데는 도움이 되었다.

### 5. 다음 타임에 바꿀 것

- 유지할 것과 그 이유
  > 테스트를 먼저 읽고 필요한 기능과 응답 형식을 먼저 파악하는 방식은 계속 유지할 생각이다.  
  > 이번에도 구현 방향을 잡는 데 가장 도움이 되었기 때문이다.

  > 작은 예제나 학습 테스트를 먼저 확인하고 미션 코드에 적용하는 방식도 유지하고 싶다.  
  > 새로운 DB 관련 문법을 바로 큰 코드에 넣는 것보다 훨씬 안정감 있게 이해할 수 있었다.

  > 부가적인 궁금증을 따로 기록하고 구현을 우선 진행하는 방식도 계속 유지하려고 한다.
  > 패키지 구조를 오래 고민하기보다 먼저 단일 컨트롤러로 동작하게 만든 뒤 필요 시 분리하는 방식은 유지한다.
- 바꿀 것과 그 이유
  > 왜 KeyHolder가 필요한가? 등의 질문을 기반으로 학습이 잘 이루어졌는지 검증한다.
  > 그 순간은 이해하고 구현했다고 하지만, 메타인지를 위해서는 직접 설명해보는 과정이 필요하다고 느끼기 때문이다.
