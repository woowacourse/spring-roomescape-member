### 2. 이번 타임의 학습 전략
- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
  - 이해한 내용을 코드 작성까지 이어지도록 한 타임에 마무리하기로 했고, 실행했다.
- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.
  - `RowMapper`라는 개념 자체를 처음 봐서 AI에게 무엇인지 질문했다.
  - "DB 행을 Java 객체로 변환하는 인터페이스"라는 설명을 들었는데,
    `jdbcTemplate.query(sql, reservationRowMapper)`에서 내가 직접 호출한 것도 아니고
    파라미터를 넘긴 것도 아닌데 어디서 순회가 일어나는지 이해가 안됐다.
  - AI에게 JdbcTemplate 내부 동작을 설명 요청했고, 콜백 패턴임을 이해했다.
    JdbcTemplate이 SQL 실행 → ResultSet 순회 → 각 행마다 `mapRow()` 호출의 흐름으로 동작한다.
  - 이어서 "함수 이름도 없는데 어떻게 `mapRow`로 연결되냐"는 의문이 생겼고,
    `@FunctionalInterface`(추상 메서드가 하나뿐인 인터페이스)에 람다를 쓰면
    Java가 자동으로 그 하나의 메서드 구현으로 판단한다는 것을 배웠다.
  - 마지막으로 `rowNum`이 쓰이지도 않는데 왜 있는지,
    파라미터 이름(`rs`)을 바꾸면 동작이 달라지는지도 확인했다.
    → `rowNum`은 인터페이스 시그니처가 고정이라 선언만 해야 하고,
    파라미터 이름은 순서·타입만 맞으면 마음대로 바꿀 수 있다.

### 3. 전략 평가
- 효과적이었던 것과 그 이유
  - "왜 작동하는지"를 납득할 때까지 계속 파고드는 방식이 효과적이었다.
    단순히 "이렇게 쓰면 된다"에서 멈추지 않고 내부 동작(콜백, FunctionalInterface)까지
    이해하게 되어서 비슷한 패턴을 다음에 만나도 응용할 수 있을 것 같다.
- 비효과적이었던 것과 그 이유
  - 질문이 너무 잘게 쪼개지다 보니 흐름을 한눈에 파악하기 어려웠다.
    처음부터 "JdbcTemplate 내부 동작 전체를 흐름도로 보여줘" 같이 큰 그림을 먼저 요청했으면
    질문 횟수를 줄일 수 있었을 것이다.
- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  - 개념과 코드 동작 모두 몰랐는데, AI에게 단계적으로 질문하며 파고드는 방식은 잘 맞았다.

    그 다음에 세부 의문을 파고드는 순서로 가면 학습 효율이 더 높아질 것입니다."

### 5. 다음 타임에 바꿀 것
- 유지할 것과 그 이유
  - 납득이 안 되면 계속 파고드는 방식. 콜백·FunctionalInterface처럼 처음에는 낯선 개념도
    결국 내부 동작까지 이해하게 되어 다음에 비슷한 패턴을 만났을 때 응용이 가능하기 때문.
- 바꿀 것과 그 이유
  - 새로운 개념을 만나면 세부 질문 전에 "전체 흐름에서 이 개념이 왜 필요한가"를 먼저 물어보기.
    이번에는 RowMapper 개념 → 순회 방식 → 람다 연결 → 파라미터 이름 순으로 잘게 쪼개다 보니
    처음부터 큰 그림을 봤으면 절반의 질문으로 같은 이해에 도달할 수 있었을 것 같다.
