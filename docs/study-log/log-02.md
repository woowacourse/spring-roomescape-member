## 1. 막힌 것의 종류

- [ ] 개념 자체를 모르겠다
- [x] **개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다** — JdbcTemplate API, KeyHolder 사용법
- [x] **코드는 돌아가는데 이게 맞는 건지 모르겠다** — `Map<String, Object>` vs DTO 도입 결정, 컨트롤러 패키지 분리 시점

2단계는 API 사용법, 3단계는 데이터 모델 의존 관계의 처리 판단이 막혔다.

## 2. 이번 타임의 학습 전략

### 2단계 — 학습 테스트 정신 유지

**v2의 9번(i+1 끊는 기준)** 강하게 적용. 호기심 많은 부분(KeyHolder 내부, ResultSet 동작)을 지금 다 파지 않고 메모만 남기고 미루기. 본 문제(2단계 통과)에 한 칸만 더 가는 깊이로 끊었다.

JdbcTemplate API는 답 받아서 적용. v1의 3번(AI 답 금지) 양보. 다만 **각 메서드(`update`, `query`, `queryForObject`)의 차이는 본인이 정리 후 확인**하는 식으로 일부 균형.

### 3단계 — 도메인 의존 관계가 본격적으로 등장

INNER JOIN, FK, `Reservation` 안에 `ReservationTime` 객체를 두는 설계까지 한 번에 받기. 다만 **DTO 도입 시점**에서 v1의 7번(깊이 파기) 적용 — "왜 `Map<String, Object>`로 받지? 이게 정말 좋을까?"

→ 결국 `ReservationRequest` DTO 도입. **본인 의문에서 출발한 리팩토링**이라 학습이 깊어졌다.

### 패키지 분리 욕망 통제 (v2의 9번)

3단계 끝나고 컨트롤러 2개가 되니 `controller/` 패키지로 묶고 싶어졌다. AI에게 물었더니 **"4단계에서 어차피 전체 구조를 다시 잡으니 지금 하면 두 번 일"** 답을 받음 → 욕망 끊고 미룸. 결과적으로 옳았다.

## 3. 전략 평가

### 효과적이었던 것
- **i+1 끊는 기준 (9번)**: KeyHolder 내부 구현, ResultSet 메타데이터 등 큰 그림에 불필요한 깊이는 메모만 남기고 4단계 후로 미룸. 본 미션이 늘어지지 않음.
- **본인 의문에서 출발한 리팩토링 (DTO)**: 가이드에 없는 작업을 본인이 발의 → 더 깊은 학습으로 이어짐.

### 비효과적이었던 것
- **여전히 답 받기 위주**: JdbcTemplate API를 학습 테스트로 먼저 손에 익히려는 계획(log-01의 다음 타임 변경)을 못 지킴. 시간 압박에 밀림.
- **결론 명시 (10번)는 부분만**: 커밋 메시지에 한 줄 결론은 박고 있지만, 별도 메모로 남기진 않음. "여기는 끝, 다음 스텝" 구조가 약함.

### 막힌 것의 종류와 전략의 궁합
"코드로 어떻게 쓰는지" 막힘에는 **답 받고 즉시 적용**이 효율적이지만, 학습 깊이가 얕아진다. 미션이 빠르게 진행돼야 하는 압력과 v1의 2번(설계 본인 머리로) 사이의 긴장이 다시 드러남.

## 4. AI 피드백

- **DTO 도입은 옳지만 응답 DTO까지는 과하다** — 도메인 ≈ 응답일 때 매핑만 늘어나고 가치 적음. 도메인이 커지면 그때.
- **DAO/Service 분리도 4단계로 미뤄라** — 미션이 그렇게 흘러가도록 설계되어 있음. 가이드 의도 따르기.
- **9번 i+1을 잘 쓰고 있다** — 호기심 생긴 깊이를 메모로 남기고 미루는 흐름이 자연스러움.

## 5. 다음 타임에 바꿀 것

### 유지할 것

- **본인 의문에서 출발한 리팩토링**: DTO 같은 본인 발의 작업이 학습 깊이를 만든다.

### 바꿀 것

- **미뤄둔 호기심 처리**: 4단계 끝나면 미뤄둔 깊이 학습(KeyHolder, RowMapper, ResultSet, JDBC 자체) 라운드 돌리기.

## 단계 전환 체크 — 2 → 3

- **이전 단계 접근법을 그대로 쓸 것인가?** 부분 변경. 2단계는 API 사용법(답 받기 효과적)이지만, 3단계는 도메인 의존 관계 판단 — 답 받기보다는 **본인의 의문에서 출발한 설계 결정**이 학습에 좋다.
- **바꾸는 이유**: 가이드도 "단계 3은 돌아가게 만든 뒤 리팩터링하는 전략"을 예시로 들었음. 일단 통과시키고, 의문이 생기면 그때 깊이 들어가기.
- **레벨1에서 발견한 "나에게 맞는 전략"이 새 단계에서도 통할까?** 통한다. 도메인 의존 관계 처리는 레벨1 OOP 미션(블랙잭, 장기)에서 이미 다뤘던 영역. 익숙한 영역이라 v1의 2번(AI 배제 설계)이 다시 작동.
