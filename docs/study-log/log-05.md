```text
3. @DirtiesContext 애노테이션을 붙이니 동시에 실행해도 테스트가 통과했다. 
4. 이와 같은 상황을 인지하고, @DirtiesContext에 대한 개념을 공부했다. 

## 목표
테스트에 @DirtiesContext 애노테이션을 붙이는 이유를 이해한다. 

## 학습 이유
여러 개의 테스트를 동시에 돌릴 때, @DirtiesContext 애노테이션을 붙이지 않으면 테스트가 실패하는 상황을 겪었다. 

## 정보 수집
- 공식 문서 - [@DirtiesContext](https://docs.spring.io/spring-framework/reference/testing/annotations/integration-spring/annotation-dirtiescontext.html)
	- 테스트 실행 중에 기본이 되는 Spring ApplicationContext가 오염되었음을 나타내고, 해당 컨텍스트는 폐기되어야 한다. 
	- 애플리케이션 컨텍스트가 오염된 것으로 표시되면 테스트 프레임워크의 캐시에서 제거되고 닫힌다. 결과적으로, 동일한 구성 메타데이터를 가진 컨텍스트가 필요한 이후의 테스트를 위해 기본 Spring 컨테이너가 다시 빌드된다. 
	- 클래스 수준과 메서드 수준 모두에서 사용할 수 있다. 
	- 클래스 모드가 BEFORE_CLASS로 설정된 클래스에 선언된 경우, 현재 테스트 실행 전에 컨텍스트를 오염시킨다.
	- 클래스 모드가 AFTER_CLASS(기본)로 설정된 클래스에 선언된 경우, 현재 테스트 실행 후 컨텍스트를 오염시킨다.
	- 클래스 모드가 BEFORE_EACH_TEST_METHOD로 설정된 클래스에 선언된 경우, 현재 테스트 클래스의 각 테스트 메서드 실행 전에 컨텍스트를 오염시킨다.
	- 클래스 모드가 AFTER_EACH_TEST_METHOD로 설정된 클래스에 선언된 경우, 현재 테스트 클래스의 각 테스트 메서드 실행 후에 컨텍스트를 오염시킨다.

- 테스트 코드에 @DirtiesContext 애노테이션을 붙이지 않았을 때 왜 테스트가 실패했을까?
	- 데이터베이스의 상태 공유 때문이다. 데이터를 추가하는 로직만 있는 테스트가 먼저 실행되고, 이후에 데이터를 추가한 후 현재 데이터가 1개인지 확인하는 테스트가 실행된다면 데이터가 2개가 되어 테스트가 실패한다.

## 나만의 언어 (구체적 예시 포함)
@DirtiesContext는 Spring ApplicationContext가 오염되었음을 나타낸다. 
테스트는 어느 테스트가 먼저 수행되느냐에 따라 성공 결과가 달라지면 안 된다.
만약 데이터베이스 관련 테스트 메서드가 2개인데 Spring ApplicationContext가 계속 유지된다면, 데이터베이스 상태가 공유되어 테스트가 실패할 확률이 발생한다. 

- 테스트 실패 예시
A. Insert 테스트
B. 한 행을 Insert한 후 전체 행이 1개인지 테스트 
ApplicationContext가 유지되는 상황에서 A가 수행되고 B가 수행된다면, Insert가 2번 발생했기 때문에 db에는 2개의 행이 존재한다. B에서는 행이 1개인지 비교하고 있는데, 실제로는 2개의 행이 존재하기 때문에 테스트가 실패한다.

## 나만의 언어 검증
- (오개념) 이 어노테이션의 본질은 Bean의 상태가 변했을 때(예: 싱글톤 빈의 필드값 변경) 컨텍스트를 새로 고치는 것이지, 단순히 DB 데이터를 지우기 위한 도구가 아니다. 지금은 너무 DB 초기화에 치중되어 있다. 
	- 지금은 내장형 H2 DB를 쓰기 때문에 DB도 같이 초기화되어 문제가 해결되는 것처럼 보이지만, 외부 실제 DB를 사용하면 @DirtiesContext로 스프링을 다시 켜도 DB에 쌓인 데이터는 지워지지 않음

## 결론
@DirtiesContext는 Spring ApplicationContext가 오염되었음을 나타낸다. 오염된 컨텍스트는 닫히고, 새로운 컨텍스트가 생성된다. 
만약 이전 테스트가 Bean의 상태를 변경했을 때 ApplicationContext가 유지된다면 다음 테스트에 영향을 줄 수 있다. 이를 방지하기 위해 클래스 단위, 메서드 단위로 컨텍스트를 오염시킬 수 있다. 

- 테스트 실패 경험
A. Insert 테스트
B. 한 행을 Insert한 후 전체 행이 1개인지 테스트 
ApplicationContext가 유지되는 상황에서 A가 수행되고 B가 수행된다면, Insert가 2번 발생했기 때문에 db에는 2개의 행이 존재한다. B에서는 행이 1개인지 비교하고 있는데, 실제로는 2개의 행이 존재하기 때문에 테스트가 실패한다.
- 내가 사용한 메모리 db는 Spring Context와 생명 주기가 같아서 컨텍스트를 오염시키고 재생성하는 과정에서 db도 초기화돼 해결된 것이다. 
- 하지만 해당 애노테이션은 단순히 db를 초기화하기 위해 사용하는 게 아니라, Bean의 상태 변경을 초기화하기 위함이란 걸 주의해야 한다. 
```
