```
# 실제 학습 과정
- 4단계의 목적은 controller가 repository를 직접 의존하지 않도록 service layer를 분리하는 것이었다.
- ReservationController에서 ReservationRepository를 직접 주입받던 구조를 ReservationService를 통해 접근하도록 변경했다.
- ReservationTimeController도 동일하게 ReservationTimeService를 추가해서 controller, service,  repository 흐름으로 변경했다.
- service layer를 분리하면서 controller 내부에서는 request를 받고, service를 호출하고, response로 변환하는 역할만 남기도록 정리했다.
- 이 과정에서 domain과 entity를 나눠야 하는지 의문이 들었다.
  특히 현재 Reservation, ReservationTime은 JPA Entity가 아니라 일반 객체이기 때문에 entity라는 이름이 적절한지 고민했다.
- 하지만 이번 단계의 목적은 계층 분리였고, domain/entity 개념을 깊게 파기 시작하면 본래 목적에서 벗어난다고 판단했다.
- 그래서 패키지명이나 domain/entity 분리 문제는 기록만 하고, service layer 분리를 먼저 마무리했다.
- 추가로 service가 ReservationResponse 같은 응답 DTO를 반환해야 하는지, 아니면 Reservation 도메인을 반환하고 controller에서 response로 변환해야 하는지도 고민했다.
- 현재 단계에서는 service가 도메인을 반환하고 controller가 response로 변환하는 구조가 더 자연스럽다고 판단했다.
  service가 HTTP 응답 구조까지 알 필요는 없다고 보았기 때문이다.
- repository의 save 반환값도 다시 정리했다.
  기존에는 save 내부에서 insert 후 findById까지 수행해서 Reservation 객체를 반환하는 구조를 생각했다.
  하지만 service layer가 생긴 상황에서는 repository.save는 저장 후 생성된 id만 반환하고, service에서 그 id로 다시 findById를 호출해 Reservation을 반환하는 흐름이 더 자연스럽다고 판단했다.
- 이렇게 바꾸니 repository는 DB 저장과 조회를 각각 담당하고, service는 "예약을 생성한 뒤 생성된 예약을 다시 조회한다"는 흐름을 담당하게 되었다.
- 또한 service 메서드 안에서 save와 findById처럼 여러 DB 작업이 이어지기 때문에, 하나의 흐름으로 보장하기 위해 트랜잭션으로 묶는 것이 적절하다고 판단했다.

# 후기
- 이번에는 이전 로그에서 정한 전략이 잘 적용됐다.
- service layer 분리 중 domain/entity, DTO 변환 위치 같은 의문이 떠올랐지만, 바로 깊게 들어가지 않았다.
- 먼저 이번 단계의 목적이 "service layer 분리"라는 것을 확인했고, 그 목적에 직접 필요한 부분만 처리했다.
- 특히 save가 객체를 반환할지 id를 반환할지 고민했는데, service layer의 역할과 연결해서 생각하니 판단 기준이 명확해졌다.
- repository가 모든 것을 완성해서 반환하기보다, service가 여러 repository 동작을 조합하는 흐름이 더 계층 분리에 맞다고 느꼈다.
```

### 3. 전략 평가

- 효과적이었던 것과 그 이유

```
# 효과적이었던 것
- 이번 단계의 목적을 먼저 확인한 것
- domain/entity 분리처럼 추가로 떠오른 의문을 바로 해결하려 하지 않고 기록만 한 것
- controller, service, repository의 책임을 현재 단계에 필요한 수준으로만 나눈 것

# 이유
- service layer 분리라는 목적이 명확했기 때문에, 불필요하게 설계 논의를 확장하지 않을 수 있었다.
- domain과 entity의 차이는 중요한 주제지만, 이번 단계에서 반드시 해결해야 하는 문제는 아니었다.
```

- 비효과적이었던 것과 그 이유

```
# 비효과적이었던 것
- service layer를 분리하면서 순간적으로 domain/entity 패키지 구조까지 같이 정리하고 싶어진 것
- 처음에는 save 메서드가 객체까지 완성해서 반환해야 한다고 생각한 것

# 이유
- 계층 분리를 하다 보면 자연스럽게 패키지 구조, 도메인 모델, DTO 책임까지 연결된다.
- 하지만 이번 단계의 핵심은 전체 아키텍처를 완성하는 것이 아니라 controller와 repository 사이에 service layer를 두는 것이었다.
- 떠오른 의문을 바로 확장했다면 학습 범위가 다시 커졌을 가능성이 있다.
```

```
# 막힌 것의 종류와 전략 궁합
```

### 4. AI 피드백

- 자신의 학습 전략에 대해 AI 학습 전문가에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록

```
# 나의 전략 분석
- 이전에는 새로운 의문이 생기면 그 의문도 함께 해결하려는 경향이 있었다.
- 이번에는 domain/entity, DTO 변환 위치 같은 의문이 생겼지만, 현재 목표와 직접 연결되는지 먼저 판단했다.
- 그 결과 service layer 분리라는 본래 목적을 유지할 수 있었다.
- 반면 save 반환값 문제는 service layer가 생기면서 직접적으로 연결되는 문제였기 때문에 현재 단계에서 정리했다.

# AI 학습 전문가의 피드백
- 이번처럼 짧은 작업에서는 "더 알고 싶은 것"과 "지금 해결해야 하는 것"을 구분하는 것이 중요하다.
- 특히 계층 분리 작업은 자연스럽게 아키텍처 전반의 의문을 불러오기 때문에, 한 번에 전부 정리하려 하면 범위가 커진다.
- 다만 현재 계층의 책임과 직접 연결되는 문제는 넘기지 말고 지금 정리하는 것이 좋다.
- save가 id를 반환할지 객체를 반환할지는 단순 구현 문제가 아니라 repository와 service의 책임 분리 문제이므로 이번 단계에서 다루는 것이 적절했다.

# 유용했던 제안
- 의문이 생겼을 때 바로 해결하지 않고 아래처럼 분류한다.
  1. 지금 구현을 막는 의문
  2. 현재 계층 분리와 직접 연결되는 의문
  3. 더 좋은 구조를 위한 의문
  4. 다음 단계에서 다시 봐도 되는 의문
- 이번에는 domain/entity 분리 문제를 4번으로 두고 넘어갔다.
- 반대로 save 반환값과 트랜잭션 문제는 service layer의 책임과 직접 연결되므로 현재 단계에서 정리했다.

# 적용 결과
- 작업 중 생긴 의문을 통제하면서 service layer 분리라는 목적을 빠르게 달성할 수 있었다.
- 이전보다 "지금 해결할 문제"와 "나중에 볼 문제"를 구분하는 감각이 좋아졌다.
- save와 findById를 service에서 하나의 흐름으로 조합하면서 service layer가 왜 필요한지 더 구체적으로 이해할 수 있었다.
```

```
# 유지할 것
- 구현 중 새로운 의문이 생기면 먼저 현재 목적과 직접 연결되는지 확인하는 방식
- 여러 주제가 동시에 떠오르면 분리하고, 지금 목표에 필요한 것부터 처리하는 방식

- repository는 DB 작업 단위에 집중하고, service에서 여러 repository 동작을 조합하는 방식

# 이유
- 이번 방식은 짧은 시간 안에 목표를 달성하는 데 효과적이었다.

- save 후 findById를 service에서 조합하면서 service layer의 역할이 더 명확해졌다.
```
