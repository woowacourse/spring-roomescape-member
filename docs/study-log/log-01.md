### 3. 전략 평가

### 단계 전환 체크

- 이전 단계에서 쓴 접근법을 그대로 쓸 것인가, 바꿔볼 것인가?
    - 완전히 바꾸기보다는, 큰 흐름은 유지하고 일부를 조정해볼 생각이다.
    - 1단계에서는 테스트를 먼저 읽고 최소 기능을 빠르게 구현하는 방식이 효과적이었기 때문에, 이 접근은 유지하려고 한다.
    - 다만 2단계에서는 DB 연동과 `JdbcTemplate` 사용법처럼 문법과 API 사용 방식이 더 중요해지므로, 구현 전에 작은 예제를 먼저 확인하는 단계를 추가해보려고 한다.

- 바꾼다면 왜? 바꾸지 않는다면 왜?
    - 1단계의 어려움은 요청-응답 흐름과 어노테이션을 실제 코드에 적용하는 것이었고, 이때는 작은 API를 빠르게 완성해보는 방식이 잘 맞았다.
    - 하지만 2단계에서는 스프링 MVC 흐름보다 `JdbcTemplate`, SQL, RowMapper 같은 구체적인 사용법에서 막힐 가능성이 크다고 느꼈다.
    - 그래서 테스트를 먼저 읽고 구현 목표를 작게 나누는 방식은 유지하되, DB 관련 문법은 바로 기능 코드에 넣기보다 작은 학습 예제나 공식 예시를 먼저 보고 적용하는 방식으로 바꾸려고 한다.

- 이전 단계에서 발견한 "나에게 맞는 전략"이 새 단계에서도 통할 것 같은가?
    - 부가적인 궁금증을 따로 기록하고 구현을 먼저 끝내는 방식은 2단계에서도 여전히 유효할 것 같다.
    - 손으로 개념을 정리하는 방식도 새로운 API 사용법을 익힐 때 도움이 될 것 같다.
    - 반면 구조를 오래 고민하는 것은 2단계에서도 효율이 떨어질 수 있어서, 우선 동작하게 만든 뒤 필요할 때 정리하는 쪽이 더 잘 맞을 것 같다.
