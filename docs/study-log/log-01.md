### 3. 전략 평가
- 효과적이었던 것과 그 이유
  - 트러블 슈팅 기록에 대한 부담과 소요시간이 줄었으므로 효과적이라고 판단했다.
- 비효과적이었던 것과 그 이유
  - 할 일(1단계)를 조회, 추가, 삭제로 목록을 나눴는데, 막상 구현하니 한번에 섞어서 몰아서 하게 되었다
  - 조회를 구현하다가 다른 걸 구현하거나, A의 구현을 마치고 B를 구현하던 도중 다시 A를 리팩터링하거나 픽스하는 등
- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  - 전략에 없던 것인데 일단 한 번 코드를 써 보는 게 좋은 것 같다(무작정 창조가 아니라, 완벽히 이론을 숙지하고 코딩할 필요가 없을 것 같다는 뜻)


### 4. AI 피드백
> 조회, 추가, 삭제를 나눴지만 결국 섞여버린 이유는, 각 작업이 '독립적인 성공(Success)'으로 정의되지 않았기 때문. 조회를 하다가 리팩터링으로 새는 것은 뇌가 '완료'라는 보상을 받지 못해 계속해서 더 큰 자극(다른 기능 구현)을 찾아 헤매는 상태.
- **가설 기반 코딩 (Hypothesis-Driven Coding)**
  >"그냥 해보는 것"에서 한 걸음 더 나아가, 코드를 치기 전 딱 5초만 '가설'을 세우고 실행

  - 방법: "직감적으로 List로 감싸보자" 대신, "List로 감싸면 Spring이 JSON 배열로 변환해줄 것이다"라고 뇌에 가설을 선언한 뒤 실행
  
- **리팩터링 '주차장' (The Refactoring Parking Lot)**
  > 구현 도중 "아, 이거 고치고 싶은데?"라는 생각이 들 때 현재의 버티컬 슬라이스를 깨지 않도록 방어

  - 방법: 구현 중 발견한 리팩터링 거리나 찜찜한 부분은 코드 옆에 // TODO: Parking 주석으로만 남기고 절대 건드리지 않는다. 
  - 현재 목표한 '최소 단위의 성공(예: 조회 API 통과)'이 끝난 후에만 건드린다.

- **할 일 분배의 재정의**
  > 로그에서 "조회, 추가, 삭제"로 나눈 것은 수평적 나열에 가깝다. 이를 v3의 버티컬 슬라이스로 더 잘게 쪼개면 이렇다.

  - "DB에서 데이터 1건을 꺼내와서 콘솔에 찍기" (성공 여부 10분 내 확인 가능)
  - "조회된 데이터를 DTO에 담아 API 응답으로 내보내기" (다음 10분 성공)
