## 1. 막힌 것의 종류

- [x] **개념 자체를 모르겠다** — `@RestController`, `@RequestBody`, `@PathVariable`이 각각 무슨 일을 하는지
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다

스프링 프로젝트의 전체 구조 자체가 처음이라, 메모리 기반 CRUD를 만들기 전에 어노테이션의 역할부터 이해가 필요했다.

## 2. 이번 타임의 학습 전략

**v1의 1번(출제자 의도)**으로 시작. "왜 데이터베이스 없이 메모리로만 시키나?" → 단계 학습 전략(환경 → API → 데이터 → 설계)의 **첫 단계는 환경/개념의 벽을 넘는 것**이라는 출제자 의도 파악.

코드 짜고 → 한 단계씩 개념 이해 → 커밋 푸시 사이클로 진행. AI에 단계별 진행 + 어노테이션 의미 설명 위임. **v1의 3번(AI 답 금지) 원칙은 적용 불가** — 처음이라 답이 필요했음.

대신 사소한 의문은 즉시 추적했다:
- `Reservation`에 기본 생성자가 왜 필요한가? → Jackson 역직렬화
- `AtomicLong`은 왜 일반 `long` 안 쓰나? → 멀티스레드 안전성

## 3. 전략 평가

### 효과적이었던 것
- **빠른 진행**: 단계 막힘 없이 1단계 완주. 출제자 의도("환경 벽 넘기")와 일치.
- **사소한 의문 즉시 추적 (v1의 7번)**: "기본 생성자 왜 필요?"를 그냥 넘기지 않은 게 다음 단계 디버깅에서 도움됐다.

### 비효과적이었던 것
- **답 받기 위주 (v1의 3번 위반)**: 미션이 처음이라 어쩔 수 없었지만, 어노테이션 동작을 본인 머리로 추측하기 전에 답을 받음.
- **24시간 후 인출(v3의 11번) 미적용**: "지금 이해했다"고 느끼지만 정말 그런지 검증 안 함.

### 막힌 것의 종류와 전략의 궁합
"개념 자체를 모르겠다"에는 **답 위주 학습이 효과적**이긴 하다. 다만 v1의 3번을 양보한 만큼 v3의 11번(24시간 인출)으로 보완해야 했는데 안 했다. 이게 다음 단계로 가면서 "정말 이해했나?" 의문으로 남는다.
