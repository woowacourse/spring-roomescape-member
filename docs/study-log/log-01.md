### 2. 이번 타임의 학습 전략

#### 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?

##### - 적용하려던 전략 (학습 방법 v3)
> 문제 인식 → 실패 경험 → 최소 학습 → 적용 → 구조화 → 설명

특히 기존 v2에서 부족했던 부분을 보완하려고 했다:

- 추상적인 개념 학습 중심
- 실제 문제 없이 시작
- 적용/피드백이 불명확

→ 이를 개선해서

- 실제 문제 기반 학습 (Problem-Driven)
- 실패 경험을 먼저 만드는 학습 (Fail First)
- 최소 학습 후 바로 적용

으로 바꾸려고 했다.

##### - 실제 실행 여부

문제 인식부터 시작했다는 점은 v3에 맞게 잘 수행됨  
하지만 “실패 경험을 의도적으로 만든 것”은 아니고  
→ 테스트를 해석하는 과정에서 자연스럽게 실패를 경험하는 형태로 진행됨

---

#### 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.

**1. 문제 인식**
- REST Assured는 왜 필요했을까?
- 없다면 뭐가 불편했을까?

단순 사용법이 아니라 등장 배경을 이해하려고 접근

---

**2. 실패 경험**
- 테스트를 보고 바로 구현하려고 시도
- 상태를 고려하지 않은 구현 (생성 후에도 데이터가 유지되지 않음)
- 타입을 잘못 이해한 구현 (JSON 구조와 맞지 않아 파싱 실패)

→ “내가 생각한 구조”로 구현 → 테스트 실패 → 문제 인식

---

**3. 최소 학습**

학습한 내용
- Spring에서 List → JSON 배열 변환 방식
- JSON 요청 바인딩 (@RequestBody)
- REST API의 기본 구조 (GET / POST / DELETE)
- REST Assured의 JSONPath (size())

문서 전체를 보는 것이 아니라  
→ “지금 막힌 문제 해결에 필요한 만큼만” 학습

**4. 적용 & 리팩토링**

### 3. 전략 평가

#### 효과적이었던 것과 그 이유

**Problem-Driven (테스트 기반 문제 인식)**

- 테스트 코드를 먼저 분석하면서 시작한 점이 가장 효과적이었다.
- “왜 필요한가?”를 따로 고민하지 않아도  
  → 테스트 자체가 요구사항을 명확하게 제공했다.

- 추상적인 개념과 달리  
  → 구체적인 입력/출력/조건이 존재해서 방향성이 명확했기 때문이다.

---

#### 비효과적이었던 것과 그 이유

**AI 활용이 정답 탐색에 가까웠다**

- “왜 틀렸는지”보다 → “어떻게 고치지?”에 집중했다.
- 빠르게 해결하려는 성향과  
  사고 과정보다 결과를 우선시 했기 때문이다.

---

#### 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?

**막힌 것의 유형**
- 요구사항 해석 문제
- 테스트가 무엇을 의미하는지 이해 부족
- 상태 관리 개념 부족
- 데이터 구조/타입 문제
- JSON ↔ Java 매핑 이해 부족

---

**잘 맞았던 부분 👍🏻**

- Problem-Driven → 요구사항 해석 문제를 해결하는 데 매우 효과적이었다.

**why?**
- 최소 학습 → 필요한 개념만 빠르게 보완했다.
