### 학습 과정

#### JPA 영속성 — 레이지 로딩

레이지 로딩은 프록시를 이용해서 실제 호출(get 등) 시점에 DB 조회를 미루는 방식.

- 장점: 불필요한 조회를 줄여 효율적
- 단점: N+1 문제 발생 가능

**트랜잭션 범위와의 관계**

- 트랜잭션 안 → 영속성 컨텍스트 살아있음 → 레이지 로딩 동작
- 트랜잭션 밖 → 영속성 컨텍스트 닫힘 → `LazyInitializationException`

**OSIV (Open Session In View)**

View까지 세션을 열어두는 패턴. Spring Boot 기본값은 `true`.

- 장점: 트랜잭션 밖에서도 레이지 로딩 가능
- 단점: DB 커넥션이 응답 반환 시까지 점유 → 트래픽 몰리면 커넥션 풀 고갈 위험
- → 실제 서비스에서는 `false` 권장

---

#### @Transactional

프록시 기술을 이용해서 메서드 실행 전후로 트랜잭션을 관리한다.
커넥션은 쓰레드에 바인딩된다.

**셀프 인보케이션**

같은 클래스 내부 메서드를 호출하면 프록시를 거치지 않아 `@Transactional` 설정이 무시된다.
`propagation` 설정도 적용 안 됨.

해결책: 별도 빈(서비스)으로 분리해서 호출

**@Async와 트랜잭션**

커넥션이 쓰레드에 바인딩되므로, `@Async`는 다른 쓰레드를 사용해 트랜잭션이 전파되지 않는다.
굳이 전파하려는 설계 자체가 잘못된 것 — 부모 트랜잭션 커밋 전에 비동기 작업이 실행되면 아직 반영되지 않은 데이터를 읽는 문제가 생긴다.

올바른 설계: 부모 트랜잭션 커밋 이후 비동기 작업 실행, 필요 시 내부에서 새 트랜잭션(`REQUIRES_NEW`) 열기

## 3. 전략 평가

- 효과적이었던 것과 그 이유
  질문을 통해 스스로 답을 찾아가는 방식이 효과적이었다.
  특히 "트랜잭션이 닫히면 세션도 닫히는가?"를 스스로 추론한 지점이 인상적이었다 — 직접 연결고리를 만들어냈다.
  `@Async`와 트랜잭션 전파 불가 문제도 "왜 전파가 안 되는지"에서 "왜 전파하면 안 되는지"로 자연스럽게 사고가 확장됐다.

- 비효과적이었던 것과 그 이유
  JPA 영속성과 @Transactional 두 주제를 한 타임에 다뤘다.
  각각의 개념이 연결되어 있어 흐름은 자연스러웠지만, 범위가 넓어서 각 개념의 깊이는 얕게 남은 느낌이다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  테코톡에서 들은 개념을 정리하는 상황이었기 때문에, 개념의 큰 그림은 있었다.
  질문을 통해 빈 곳을 채워가는 전략이 이 상황과 잘 맞았다.

## 4. AI 피드백

오늘 학습에서 주목할 점:
레이지 로딩 → `LazyInitializationException` → OSIV, 셀프 인보케이션 → `@Async` 트랜잭션 전파 모두 "프록시와 트랜잭션 범위"라는 하나의 축으로 연결된다.
개별 개념을 외우는 것보다 이 축을 중심으로 연결해두면 다음 사이클에서 더 빠르게 깊이를 더할 수 있다.

1. **트랜잭션 격리 수준** — `isolation` 옵션. REPEATABLE_READ, SERIALIZABLE이 각각 어떤 문제를 막는지
2. **readOnly = true의 실제 효과** — 단순히 쓰기 막는 것 이상으로 어떤 최적화가 생기는지
3. **checked vs unchecked 예외와 롤백** — 기본적으로 `RuntimeException`만 롤백되는데, checked 예외는 어떻게 처리해야 하는지
