### 14. 익숙함을 파고들기

> 리뷰 포인트는 틀린 코드를 찾는 데서만 나오는 게 아니라, 당연하게 선택한 설계의 기준을 다시 묻는 데서도 나올 수 있습니다.  
> 내가 당연하게 선택한 것을 몇 가지 적고, 다른 선택지도 함께 비교해보면 좋은 학습이 될 것 같아요.

- 의도/기준/생각
    - 테스트 문법 / 계층 분리 제외곤 경험한 내용 기반으로 선택한 부분이 많았다
    - 무결성 검증 / 업데이트를 위한 도메인 수정(PATCH) 를 서비스가?
        - 도메인이 스스로 자신의 무결성에 대한 규칙을 보유하고 행동해야
        - 서비스가 도메인의 조립 책임까지 가지면 변경의 전파가 심해지고 TDA 위반
    - 전역 예외 핸들링에서 메시지만 재정의?
        - 아무튼 커스텀 예외를 썼었다
        - 왜? 어떤 예외인지 명확히 표현해야 좋은 에러 응답을 내려줄 수 있어서
    - 도메인 객체 엔티티는 불변 객체?
        - 상태가 변해도(new Reservation) 식별자로 비교하면 엔티티로서 기능할 수 있지 않나 `record`
        - 아니다. 동등성 비교가 자동으로 정의되고 값의 set 자체를 막아버리는 것은 바람직하지 않다. `class`
    - 그 외 기타등등
