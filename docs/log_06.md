### 학습 과정

#### MVCC (Multi-Version Concurrency Control)

트랜잭션이 시작된 시점의 스냅샷을 찍어두고, 다른 트랜잭션이 중간에 값을 바꿔도 내 트랜잭션은 처음 찍은 스냅샷을 읽는다. 이 스냅샷이 "버전"이다.

```
트랜잭션 A: 조회 → 스냅샷(version 1) 생성
트랜잭션 B: 같은 행 수정 후 커밋 → version 2
트랜잭션 A: 다시 조회 → 여전히 version 1을 읽음 (Non-Repeatable Read 방지)
```

**장점**: 읽기에 락이 없어서 읽기와 쓰기가 서로를 막지 않는다 → 성능 유리

**단점**:
- 트랜잭션이 길게 유지되면 오래된 버전(Undo Log)이 쌓여 스토리지 부담
- 사용이 끝난 버전을 주기적으로 정리(Purge)하는 비용 발생
- 두 트랜잭션이 같은 행을 동시에 수정하려 할 때 버전 충돌 발생

---

#### 낙관적 락 (Optimistic Lock)

"충돌이 드물 것이다"라고 낙관적으로 가정하고, 충돌이 발생했을 때 버전으로 감지한다.

```
트랜잭션 A: 행 읽기 → version = 1
트랜잭션 B: 행 읽기 → version = 1
트랜잭션 A: 수정 → version 1 맞음 → 성공, version = 2
트랜잭션 B: 수정 → version 1 확인 → 이미 2로 바뀜 → OptimisticLockException
```

Spring JPA에서는 `@Version` 어노테이션으로 구현. MVCC가 이미 버전을 관리하므로 Repeatable Read(3단계)와 자연스럽게 어울린다.

---

#### 비관적 락 (Pessimistic Lock)

"충돌이 자주 발생할 것이다"라고 가정하고, 처음부터 `SELECT FOR UPDATE`로 락을 걸어 다른 트랜잭션의 수정을 대기시킨다. 조회 자체는 MVCC 덕분에 여전히 가능하다.

---

#### 선택 기준

| | 낙관적 락 | 비관적 락 |
|---|---|---|
| 적합한 상황 | 충돌이 드문 경우 | 충돌이 잦은 경우 |
| 장점 | 락 없음 → 성능 유리 | 순서 보장 → 정합성 유리 |
| 단점 | 충돌 시 재시도 필요 | 락으로 인한 성능 저하 |

---

#### 오늘의 연결고리

**격리 수준 → MVCC → 낙관적 락 → 비관적 락 선택 기준**

모두 "동시성 문제를 어떤 비용으로 해결할 것인가"라는 하나의 축으로 연결된다.

## 3. 전략 평가

- 효과적이었던 것과 그 이유
  로그 작성 전 스스로 먼저 정리한 것이 효과적이었다.
  MVCC 이름을 "모델 버전"으로 잘못 기억하고 있었고, 비관적 락이 "조회를 못하게 한다"는 오개념도 스스로 정리하면서 발견했다. AI 정리를 먼저 봤다면 그냥 넘어갔을 부분들이다.

- 비효과적이었던 것과 그 이유
  낙관적 락과 비관적 락의 실제 코드 레벨 구현까지는 다루지 않았다. 개념은 잡혔지만 실제 `@Version` 사용 시 어떻게 동작하는지는 다음 사이클에서 코드로 확인이 필요하다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  개념을 처음 접하는 상황에서 추측 → 연결 질문 방식이 잘 맞았다. 특히 격리 수준에서 시작해서 자연스럽게 MVCC → 낙관적 락 → 비관적 락까지 이어진 흐름이 인상적이었다.
