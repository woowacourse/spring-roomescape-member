### 학습 과정

#### 의문의 시작

기존 `ErrorMessage`는 `message` 하나만 담고 있었다.
개발 중 에러가 났을 때 메시지만으로는 판단이 어렵고, IDE에서 다시 확인해야 했다.
→ 에러 응답에 어떤 정보가 있어야 IDE 없이도 판단할 수 있을까?

필요한 정보: 어느 EndPoint인지, 어디에서, 어떤 이유로, 에러가 났을 때 값이 뭔지

#### RFC 7807 / ProblemDetail

Spring 6.0부터 RFC 7807 스펙을 구현한 `ProblemDetail` 클래스를 제공한다.

| 필드 | 역할 |
|---|---|
| `type` | 문제 유형의 문서 URI. 기본값 `about:blank` |
| `title` | 문제 유형 요약 — 같은 유형이면 항상 같은 값 |
| `status` | HTTP 상태 코드 |
| `detail` | 이번 발생의 구체적 설명 — **반드시 String** |
| `instance` | 에러가 발생한 리소스 URI (EndPoint) |

#### detail은 반드시 String

RFC 7807에서 `detail`은 human-readable 문자열로 정의됨.
파싱하기 쉬운 구조화된 데이터가 필요하면 **extension 필드**를 추가한다.

```json
{
    "type": "about:blank",
    "title": "입력값 검증 실패",
    "status": 400,
    "detail": "하나 이상의 필드가 유효하지 않습니다.",
    "instance": "/reservations",
    "invalid-params": [
        { "field": "date", "reason": "must not be null" }
    ]
}
```

`invalid-params`가 RFC extension 필드의 예 — 클라이언트가 파싱하기 쉬운 구조화 데이터.

#### ResponseEntityExceptionHandler 상속 여부

Spring이 제공하는 `ResponseEntityExceptionHandler`를 상속하면 Spring 내부 예외들을 기본 처리해줌.
하지만 응답 형식을 완전히 제어하려면 거의 모든 메서드를 오버라이드해야 함.

→ 현재 구현은 직접 작성. Spring 내부 예외 일부를 놓칠 수 있다는 트레이드오프 존재.

## 3. 전략 평가

- 효과적이었던 것과 그 이유
  RFC 공식 문서를 기반으로 각 필드의 의미를 먼저 이해하고 구현한 게 효과적이었다.
  "detail에 Map 형식으로"라는 시도가 틀렸지만, 그 과정에서 detail이 String이어야 하는 이유를 더 명확히 이해했다.

- 비효과적이었던 것과 그 이유
  처음에는 필요성을 충분히 느끼지 못한 채 구현부터 시작했다. 경험 기반으로 필요성을 먼저 짚은 뒤에 학습 흐름이 더 잘 잡혔다.

- 막힌 것의 종류(1번)와 전략의 궁합은 어땠는가?
  공식 문서 + 직접 구현 조합이 잘 맞았다. 이름만 보고 추측 → 공식 문서로 검증하는 흐름이 효과적이었다.

## 4. AI 피드백

오늘 가장 중요한 발견은 "detail은 String, 구조화 데이터는 extension"이라는 RFC 설계 의도를 직접 부딪혀서 알게 됐다는 점이다.
처음부터 정답을 알려줬다면 그냥 넘어갔을 내용이지만, "Map 형식으로 넣고 싶다"는 시도가 있었기 때문에 RFC의 이유가 기억에 남는다.

## 5. 다음 타임에 바꿀 것

- 유지할 것과 그 이유
  개념을 설명할 때 "왜"까지 연결하기 — 오늘 `detail`이 String인 이유를 RFC에서 직접 확인하는 과정이 잘 작동했다.

- 바꿀 것과 그 이유
  리뷰 피드백 같은 외부 주제를 학습할 때도 "이게 끝나면 다음에 뭐가 궁금하지?"를 의식적으로 만들기.
  오늘은 피드백을 따라가다 보니 다음 의문을 스스로 만들 틈이 없었다 — 학습이 수동적이 됐다는 신호다.
  주도적인 사이클을 유지하려면 외부에서 주어진 주제라도 끝에 "내 의문"을 하나 만드는 연습이 필요하다.
